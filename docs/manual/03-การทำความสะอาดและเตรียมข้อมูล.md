# 03 การทำความสะอาดและเตรียมข้อมูล

ไฟล์นี้ใช้วางแนวทางการทำความสะอาดข้อมูลจาก Excel, CSV, Google Sheet หรือผลลัพธ์จากระบบ Data Request ก่อนนำเข้า Supabase และนำไปใช้ใน Dashboard, Search, Map และ AI Chatbot ของระบบ `npt_dashboard`

บทนี้ต่อจากบท 02 ที่เน้นการรวบรวมข้อมูลจากจังหวัดและอำเภอ เมื่อได้ไฟล์ต้นทางแล้ว ขั้นตอนนี้คือการปรับข้อมูลให้อยู่ในรูปแบบที่ระบบอ่านได้ ตรวจสอบได้ และนำเข้าได้โดยลดปัญหาข้อมูลเพี้ยนภายหลัง

## 1. เป้าหมายของขั้นตอนนี้

การทำความสะอาดข้อมูลมีเป้าหมายเพื่อให้ข้อมูลต้นทางกลายเป็นข้อมูลพร้อมใช้ ไม่ใช่แค่เปิดไฟล์ได้ แต่ต้องนำไปคำนวณ ค้นหา แสดงแผนที่ และตอบคำถามด้วย AI ได้อย่างน่าเชื่อถือ

เป้าหมายหลัก ได้แก่

- ทำให้ข้อมูลอยู่ในรูปแบบตารางที่อ่านได้ง่าย
- แก้ปัญหาหัวตารางไม่ชัด เซลล์รวม แถวว่าง และแถวสรุปรวม
- ทำให้ชื่ออำเภอ ตำบล วันที่ ตัวเลข และพิกัดอยู่ในรูปแบบเดียวกัน
- แยกข้อมูลส่วนบุคคลหรือข้อมูลที่ไม่ควรเผยแพร่
- เตรียมไฟล์ CSV แบบ UTF-8 สำหรับ import เข้า Data Management
- เตรียม mapping ระหว่างหัวคอลัมน์ในไฟล์กับชื่อ field ในฐานข้อมูล
- ลดความผิดพลาดก่อนนำข้อมูลเข้า Supabase

## 2. หลักคิดสำคัญ

ก่อนเริ่มแก้ไฟล์ ให้แยกความหมายของไฟล์ออกเป็น 3 ระดับ

| ระดับไฟล์ | ความหมาย | ควรทำอย่างไร |
|---|---|---|
| ไฟล์ต้นฉบับ | ไฟล์ที่ได้รับจากเจ้าของข้อมูล | เก็บไว้เสมอ ห้ามแก้ทับ |
| ไฟล์ทำงาน | ไฟล์ที่ใช้ clean และปรับรูปแบบ | แก้ไขได้ ใส่สูตรได้ ใช้ตรวจข้อมูลได้ |
| ไฟล์พร้อมนำเข้า | ไฟล์สุดท้ายที่แปลงเป็น CSV หรือรูปแบบที่ระบบใช้ | ต้องสะอาด เหลือเฉพาะคอลัมน์ที่ใช้จริง |

แนวทางที่แนะนำ:

1. สำรองไฟล์ต้นฉบับก่อนเสมอ
2. สร้างไฟล์ทำงานอีกชุดหนึ่งสำหรับแก้ไข
3. อย่าลบข้อมูลที่ยังไม่แน่ใจ ให้ย้ายไป sheet หมายเหตุแทน
4. บันทึกสิ่งที่แก้ไว้ใน changelog สั้น ๆ
5. ตรวจซ้ำก่อน export เป็น CSV

## 3. รูปแบบตารางที่ระบบต้องการ

ข้อมูลที่พร้อมนำเข้าระบบควรเป็นตารางแบบปกติ

ลักษณะที่ควรมี:

- แถวแรกเป็นหัวคอลัมน์
- หนึ่งแถวแทนหนึ่งรายการข้อมูล
- หนึ่งคอลัมน์แทนหนึ่งชนิดข้อมูล
- ไม่มีการรวมเซลล์ในพื้นที่ข้อมูล
- ไม่มีหัวเรื่องหลายบรรทัดปนกับตาราง
- ไม่มีแถวสรุปรวมอยู่ในข้อมูลหลัก
- ไม่มีหมายเหตุยาว ๆ แทรกกลางตาราง
- ไม่มีรูปภาพ กล่องข้อความ หรือ object ที่ระบบอ่านไม่ได้

ตัวอย่างรูปแบบที่ควรแก้:

| ปัญหา | วิธีแก้ |
|---|---|
| มีหัวรายงาน 3-5 แถวก่อนหัวตาราง | ลบหรือย้ายไป sheet รายละเอียดไฟล์ |
| มีเซลล์รวมในชื่ออำเภอ | เติมชื่ออำเภอซ้ำลงทุกแถว |
| มีแถวรวมท้ายตาราง | ลบออกจากไฟล์นำเข้า หรือแยกไปอีก sheet |
| มีหลายตารางใน sheet เดียว | แยกเป็น sheet หรือไฟล์ละ 1 ตาราง |
| มีหน่วยอยู่ในช่องตัวเลข เช่น `25 ไร่` | แยกเป็น `25` และคอลัมน์หน่วยเป็น `ไร่` |
| มีวันที่หลายรูปแบบ | แปลงเป็นรูปแบบเดียว เช่น `2026-05-10` |

## 4. ตั้งชื่อคอลัมน์ให้พร้อมใช้

ชื่อคอลัมน์มีผลต่อการ map เข้า field ของฐานข้อมูล ถ้าตั้งชื่อสม่ำเสมอ ระบบจะ import และตรวจข้อมูลได้ง่ายขึ้น

แนวทางตั้งชื่อคอลัมน์:

- ใช้ชื่อสั้น ชัดเจน
- หลีกเลี่ยงการขึ้นบรรทัดใหม่ในหัวคอลัมน์
- หลีกเลี่ยงเครื่องหมายพิเศษที่ไม่จำเป็น
- ถ้าเตรียมสำหรับระบบโดยตรง ควรใช้ชื่อ field ภาษาอังกฤษแบบ snake_case
- ถ้าใช้หัวคอลัมน์ภาษาไทย ต้องเตรียม mapping ไปยัง field ของระบบ

ตัวอย่างการตั้งชื่อ:

| หัวคอลัมน์ในไฟล์เดิม | ชื่อที่แนะนำสำหรับไฟล์ clean | field ในระบบ |
|---|---|---|
| อำเภอ | district | `district` |
| ตำบล | subdistrict | `subdistrict` |
| ชื่อแปลงใหญ่ | plot_name | `plot_name` |
| ชนิดสินค้า | product_type | `product_type` |
| จำนวนสมาชิก | member_count | `member_count` |
| พื้นที่รวม | area_rai | `area_rai` |
| ละติจูด | latitude | `latitude` |
| ลองจิจูด | longitude | `longitude` |
| ปีงบประมาณ | fiscal_year | `fiscal_year` |
| วันที่ปรับปรุง | updated_data_at | `updated_data_at` |

หมายเหตุ: ในระบบ Import CSV ปัจจุบัน ผู้ใช้งานสามารถเลือก map คอลัมน์ CSV ไปยัง field ของตารางได้ในหน้าจอนำเข้า แต่ถ้าชื่อคอลัมน์ตรงกับ field หรือชื่อแสดงผลของระบบ จะช่วยลดงาน map ลงมาก

## 5. การจัดการชื่ออำเภอและตำบล

ชื่อพื้นที่เป็นข้อมูลสำคัญมาก เพราะใช้รวมกราฟ แสดงแผนที่ ค้นหา และสรุปข้อมูลด้วย AI

หลักการ:

- ใช้ชื่ออำเภอแบบเดียวกันทุกไฟล์
- ไม่ใส่คำว่า `อ.` หรือ `อำเภอ` ปนในบางแถว
- ไม่เว้นวรรคหน้าหลังชื่อพื้นที่
- ตรวจสะกดซ้ำ เช่น `กำแพงแสน` ต้องสะกดเหมือนกันทุกชุดข้อมูล
- ถ้าข้อมูลระดับตำบล ให้ใช้ชื่อตำบลทางราชการและแยกคอลัมน์จากอำเภอ

ตัวอย่างรูปแบบที่ควรใช้:

| ไม่ควรใช้ | ควรใช้ |
|---|---|
| อ.เมืองนครปฐม | เมืองนครปฐม |
| เมือง นครปฐม | เมืองนครปฐม |
| กำแพงแสน  | กำแพงแสน |
| บางเลน / บางเลนฯ | บางเลน |

สำหรับจังหวัดนครปฐม ควรกำหนดรายชื่ออำเภอมาตรฐานไว้ก่อน เช่น

```text
เมืองนครปฐม
กำแพงแสน
นครชัยศรี
ดอนตูม
บางเลน
สามพราน
พุทธมณฑล
```

ถ้าใช้ระบบ Data Request ควรใช้ช่องแบบ select สำหรับอำเภอ เพื่อลดปัญหาการสะกดไม่ตรงกัน

## 6. การจัดการวันที่และปีข้อมูล

วันที่ควรอยู่ในรูปแบบเดียวกันทั้งไฟล์ เพราะมีผลต่อการกรองข้อมูลและการแสดงผลใน Dashboard

รูปแบบที่แนะนำ:

| ประเภทข้อมูล | รูปแบบที่แนะนำ | ตัวอย่าง |
|---|---|---|
| วันที่ | `YYYY-MM-DD` | `2026-05-10` |
| เดือน | `YYYY-MM` | `2026-05` |
| ปี พ.ศ. | ตัวเลข 4 หลัก | `2569` |
| ปีงบประมาณ | ตัวเลข 4 หลัก | `2569` |
| วันที่และเวลา | `YYYY-MM-DD HH:mm:ss` | `2026-05-10 14:30:00` |

ข้อควรระวัง:

- อย่าปน `10/05/2569`, `5/10/2026`, `10 พ.ค. 69` ในคอลัมน์เดียวกัน
- ถ้าเป็นวันที่ พ.ศ. ให้แปลงหรือระบุให้ชัดก่อนนำเข้าระบบ
- ถ้าไม่ทราบวันจริง แต่ทราบแค่เดือนหรือปี ให้แยกคอลัมน์เป็น `month` หรือ `year` แทนการใส่วันที่สมมติ
- วันที่ปรับปรุงข้อมูลควรแยกจากวันที่เกิดเหตุการณ์

## 7. การจัดการตัวเลข

ตัวเลขที่พร้อมนำเข้าระบบควรเป็นตัวเลขล้วน เพื่อให้ระบบรวมผล คำนวณค่าเฉลี่ย จัดอันดับ และสร้างกราฟได้

หลักการ:

- ไม่ใส่หน่วยในช่องตัวเลข
- ไม่ใช้เครื่องหมาย `-` แทนค่าว่าง ถ้าต้องการค่าว่างให้ปล่อยว่างหรือใช้ 0 ตามความหมายจริง
- ไม่ใส่ comma คั่นหลักพัน ถ้าระบบนำเข้าแล้วอ่านเป็นข้อความ
- ตรวจเลขไทยและเลขอารบิกให้สม่ำเสมอ
- แยกจำนวน พื้นที่ และงบประมาณเป็นคนละคอลัมน์

ตัวอย่าง:

| ค่าเดิม | ค่า clean | หมายเหตุ |
|---|---:|---|
| 1,250 ไร่ | 1250 | แยกหน่วยไปคอลัมน์ `area_unit` |
| 20 ราย | 20 | ใช้คอลัมน์ `member_count` |
| - | | ถ้าไม่ทราบค่าให้เว้นว่าง |
| ๕๐ | 50 | ใช้เลขอารบิก |
| 10 ไร่ 2 งาน | 10.5 | ถ้าต้องแปลงเป็นไร่ ต้องตกลงสูตรก่อน |

ถ้าเป็นงบประมาณ ควรเก็บเป็นตัวเลขหน่วยบาท เช่น `350000` ไม่ควรเก็บเป็น `350,000 บาท`

## 8. การจัดการพิกัดและข้อมูลแผนที่

ข้อมูลที่ใช้กับแผนที่ควรมีพิกัดที่ตรวจแล้ว เพราะพิกัดผิดจะทำให้ marker ไปผิดจังหวัดหรือแสดงผลไม่ได้

คอลัมน์ที่แนะนำ:

| คอลัมน์ | ความหมาย | ตัวอย่าง |
|---|---|---|
| `latitude` | ละติจูด | `13.8196` |
| `longitude` | ลองจิจูด | `100.0443` |
| `location_name` | ชื่อสถานที่ | `ศูนย์เรียนรู้...` |
| `address` | ที่อยู่โดยย่อ | `ต.บางกระทึก อ.สามพราน` |

แนวทางตรวจพิกัด:

- พิกัดในประเทศไทยโดยทั่วไป latitude ประมาณ 5-21 และ longitude ประมาณ 97-106
- จังหวัดนครปฐมควรอยู่ใกล้ช่วง latitude ประมาณ 13.x และ longitude ประมาณ 99.x-100.x
- ห้ามสลับ latitude กับ longitude
- ถ้าไม่มีพิกัด ไม่ควรเดาเอง ควรใช้ข้อมูลจากเจ้าของพื้นที่หรือแหล่งที่ตรวจได้
- ถ้าพิกัดเป็น DMS เช่น องศา ลิปดา ฟิลิปดา ต้องแปลงเป็น decimal degree ก่อน

## 9. การจัดการค่าว่างและค่าที่ไม่ทราบ

ค่าว่างควรมีความหมายชัดเจน ไม่ควรใช้สัญลักษณ์หลายแบบในคอลัมน์เดียวกัน

| สถานการณ์ | วิธีใส่ค่าที่แนะนำ |
|---|---|
| ไม่มีข้อมูลจริง | เว้นว่าง |
| ค่าเป็นศูนย์จริง | `0` |
| ไม่เกี่ยวข้องกับรายการนั้น | เว้นว่าง หรือใช้ `not_applicable` ถ้าระบบต้องแยกความหมาย |
| อยู่ระหว่างตรวจสอบ | ใช้คอลัมน์สถานะ เช่น `pending_review` |

ไม่ควรใช้ปนกันหลายแบบ เช่น `-`, `ไม่มี`, `n/a`, `ไม่ระบุ`, `0` ถ้ายังไม่ได้ตกลงความหมาย เพราะระบบอาจตีความผิด

## 10. การตรวจข้อมูลซ้ำ

ข้อมูลซ้ำทำให้ Dashboard นับจำนวนเกินจริง และทำให้ AI ตอบตัวเลขผิด

ควรกำหนด key สำหรับตรวจซ้ำของแต่ละชุดข้อมูล เช่น

| ชุดข้อมูล | key ที่ใช้ตรวจซ้ำได้ |
|---|---|
| แปลงใหญ่ | ชื่อแปลง + อำเภอ + ปีข้อมูล |
| วิสาหกิจชุมชน | ชื่อวิสาหกิจ + อำเภอ หรือเลขทะเบียนถ้ามี |
| ศูนย์เรียนรู้ | ชื่อศูนย์ + อำเภอ |
| งบประมาณ | โครงการ + ปีงบประมาณ + รหัสกิจกรรม |
| จุดความร้อน | วันที่ + พิกัด + แหล่งข้อมูล |
| คำขอข้อมูลจากอำเภอ | request_id + district |

ขั้นตอนตรวจซ้ำ:

1. สร้างคอลัมน์ช่วย เช่น `duplicate_key`
2. รวมค่าที่ใช้ตรวจ เช่น ชื่อ + อำเภอ + ปี
3. ใช้ filter หรือ conditional formatting หา key ที่ซ้ำ
4. ตรวจว่าเป็นข้อมูลซ้ำจริง หรือเป็นคนละรายการที่ชื่อคล้ายกัน
5. ลบหรือรวมรายการซ้ำโดยบันทึกหมายเหตุไว้

## 11. การแยกข้อมูลส่วนบุคคลและข้อมูลภายใน

ก่อนนำข้อมูลเข้า Dashboard ต้องแยก field ที่เผยแพร่ได้กับ field ที่ควรใช้เฉพาะภายใน

ตัวอย่างข้อมูลที่ต้องระวัง:

- เลขบัตรประชาชน
- เบอร์โทรศัพท์ส่วนตัว
- ที่อยู่ละเอียดรายบุคคล
- วันเดือนปีเกิด
- เลขบัญชีธนาคาร
- ข้อมูลรายได้หรือหนี้สินรายบุคคล
- รายชื่อเกษตรกรพร้อมข้อมูลติดต่อ

แนวทางจัดการ:

- ถ้าใช้หน้า public ให้แสดงเป็นข้อมูลรวมระดับอำเภอ/ตำบลแทนข้อมูลรายบุคคล
- ถ้าจำเป็นต้องเก็บข้อมูลส่วนบุคคล ให้จำกัดสิทธิ์ในระบบและตรวจ RLS ใน Supabase
- แยกคอลัมน์ที่ใช้ภายในออกจากไฟล์ public
- ไม่วางไฟล์ต้นทางที่มีข้อมูลส่วนบุคคลไว้ใน public repository
- ถ้าสงสัยว่าเผยแพร่ได้หรือไม่ ให้จัดเป็น internal ก่อน

## 12. เตรียมไฟล์สำหรับ Import CSV

ระบบ Data Management ปัจจุบันรองรับการ import จาก CSV ผ่านปุ่ม `Import CSV` ในตารางข้อมูล โดยมีขั้นตอนเลือกไฟล์ preview ข้อมูล map คอลัมน์ และ insert เข้า Supabase

ข้อกำหนดไฟล์ CSV:

- ไฟล์ควรเป็น UTF-8
- แถวแรกต้องเป็น header
- รองรับตัวคั่น comma `,` หรือ semicolon `;`
- คอลัมน์ที่ไม่ต้องการนำเข้า สามารถข้ามตอน map ได้
- ระบบจะนำเข้าเป็นชุด ชุดละประมาณ 50 แถว
- การ import เป็นการ insert ข้อมูลใหม่ ควรตรวจข้อมูลซ้ำก่อนนำเข้า

ขั้นตอนเตรียมไฟล์:

1. ทำข้อมูลใน Excel หรือ Google Sheet ให้สะอาดก่อน
2. ลบแถวหัวรายงาน แถวว่าง และแถวรวม
3. ตรวจชนิดข้อมูล เช่น วันที่ ตัวเลข พิกัด
4. ตั้งชื่อ header ให้ตรงกับ field หรือเตรียม mapping ไว้
5. Export หรือ Save as เป็น CSV UTF-8
6. เปิด CSV ตรวจอีกครั้งว่าภาษาไทยไม่เพี้ยน
7. นำเข้าในระบบและตรวจ preview ก่อนกด import

## 13. การ map คอลัมน์ตอนนำเข้า

เมื่อ import CSV ระบบจะแสดงหัวคอลัมน์จากไฟล์ และให้เลือก field ปลายทางของฐานข้อมูล

แนวทาง map:

| คอลัมน์ CSV | field ปลายทาง | หมายเหตุ |
|---|---|---|
| district | `district` | ชื่ออำเภอต้องตรงมาตรฐาน |
| subdistrict | `subdistrict` | ถ้ามีข้อมูลระดับตำบล |
| name | field ชื่อเฉพาะของตาราง | เช่น `plot_name`, `enterprise_name`, `center_name` |
| area_rai | field พื้นที่ของตาราง | ต้องเป็นตัวเลข |
| member_count | field จำนวนสมาชิก | ต้องเป็นตัวเลข |
| latitude | `latitude` | ถ้าตารางรองรับพิกัด |
| longitude | `longitude` | ถ้าตารางรองรับพิกัด |
| fiscal_year | `fiscal_year` | ใช้ปีงบประมาณ |
| source_file | field หมายเหตุ/แหล่งข้อมูล | ใช้ตรวจย้อนกลับ |

ข้อควรระวัง:

- อย่า map คอลัมน์เดียวกันไปหลาย field ถ้าไม่จำเป็น
- คอลัมน์ที่ไม่มั่นใจ ให้ข้ามก่อน แล้วค่อยนำเข้าเพิ่มเมื่อพร้อม
- ถ้า field ปลายทางเป็นตัวเลข แต่ข้อมูลในไฟล์เป็นข้อความ ระบบอาจนำเข้าไม่สำเร็จ
- ถ้า Supabase แจ้ง error ให้ดูช่วงแถวที่ระบบรายงาน แล้วกลับไปแก้ CSV

## 14. การเตรียมข้อมูลจาก Data Request

ข้อมูลจากระบบ Data Request จะได้ผลลัพธ์ตาม schema ที่จังหวัดกำหนด และคำตอบจากแต่ละอำเภอจะถูกรวมไว้ตาม request

แนวทางหลังปิดรับข้อมูล:

1. export ผลลัพธ์คำตอบจากระบบ
2. ตรวจว่าอำเภอส่งครบหรือไม่
3. ตรวจคอลัมน์ตาม schema ที่กำหนดไว้
4. ตรวจว่าคำตอบเป็นแถวข้อมูลจริง ไม่ใช่ข้อความอธิบายยาวในช่องเดียว
5. รวมข้อมูลจากทุกอำเภอเป็นตารางเดียว
6. clean ชื่ออำเภอ ตำบล วันที่ ตัวเลข และพิกัด
7. map เข้าตารางเป้าหมาย เช่น `learning_centers`, `agri_tourism`, `pest_centers`
8. export เป็น CSV UTF-8 สำหรับ import เข้าตารางจริง

ถ้าคำขอข้อมูลมีช่อง `ผู้ประสานงาน`, `เบอร์โทร`, หรือรายละเอียดส่วนบุคคล ควรแยกเก็บเป็นข้อมูล internal ไม่ควรใช้ในหน้า public โดยตรง

## 15. การตรวจคุณภาพข้อมูลก่อนนำเข้า

ก่อน import ควรตรวจอย่างน้อย 4 ด้าน

### 15.1 ความครบถ้วน

- [ ] มีข้อมูลในคอลัมน์จำเป็นครบ
- [ ] ไม่มีแถวว่างปนในข้อมูลหลัก
- [ ] ไม่มีรายการที่ขาดชื่ออำเภอหรือปีข้อมูล
- [ ] ถ้าใช้แผนที่ มีพิกัดในรายการที่ต้องแสดงแผนที่

### 15.2 ความถูกต้อง

- [ ] ชื่ออำเภอและตำบลสะกดตรงมาตรฐาน
- [ ] วันที่อยู่ในรูปแบบเดียวกัน
- [ ] ตัวเลขไม่มีหน่วยปนอยู่
- [ ] พิกัดไม่สลับ latitude/longitude
- [ ] ข้อมูลไม่หลุดพื้นที่จังหวัด

### 15.3 ความซ้ำ

- [ ] ตรวจรายการซ้ำด้วย key ที่กำหนดแล้ว
- [ ] แยกรายการซ้ำจริงกับรายการชื่อคล้ายกันแล้ว
- [ ] ไม่มีแถวรวมที่ทำให้ยอดนับซ้ำ

### 15.4 ความปลอดภัย

- [ ] แยกข้อมูล public/internal/restricted แล้ว
- [ ] ลบหรือซ่อนข้อมูลส่วนบุคคลจากไฟล์ public แล้ว
- [ ] ระบุแหล่งข้อมูลและวันที่ปรับปรุงแล้ว
- [ ] ไม่ใช้ไฟล์ต้นทางที่มีข้อมูลอ่อนไหวในที่สาธารณะ

## 16. ตัวอย่าง workflow การ clean ข้อมูลแปลงใหญ่

ตัวอย่างนี้ใช้กับข้อมูล `large_plots`

1. รับไฟล์ Excel จากกลุ่มงานส่งเสริมและพัฒนาการผลิต
2. สำรองไฟล์ต้นฉบับ
3. ลบหัวรายงานและแถวรวมท้ายตาราง
4. เติมชื่ออำเภอในทุกแถว ถ้าไฟล์เดิมใช้เซลล์รวม
5. แยกคอลัมน์ชื่อแปลง สินค้า พื้นที่ สมาชิก ปีข้อมูล
6. แปลงพื้นที่เป็นตัวเลขหน่วยไร่
7. ตรวจชื่ออำเภอให้ตรงมาตรฐานจังหวัด
8. ตรวจรายการซ้ำด้วย `ชื่อแปลง + อำเภอ + ปีข้อมูล`
9. แยกข้อมูลติดต่อส่วนบุคคลออกถ้าไม่จำเป็นต่อ Dashboard
10. ตั้งชื่อคอลัมน์ให้ map กับ field ของระบบได้ง่าย
11. export เป็น CSV UTF-8
12. import ผ่านหน้า Data Management ของตารางแปลงใหญ่
13. ตรวจ Dashboard และ Search ว่าข้อมูลแสดงผลถูกต้อง

## 17. ตัวอย่าง workflow การ clean ข้อมูลจุดแผนที่

ใช้กับข้อมูลศูนย์เรียนรู้ ท่องเที่ยวเกษตร ศจช. ศดปช. หรือจุดเฝ้าระวัง

1. ตรวจว่าหนึ่งแถวคือหนึ่งสถานที่หรือหนึ่งจุดข้อมูล
2. แยกชื่อสถานที่ อำเภอ ตำบล ประเภท และรายละเอียดออกเป็นคอลัมน์
3. ตรวจพิกัด latitude/longitude
4. ถ้าไม่มีพิกัด ให้ทำเครื่องหมายว่ายังไม่พร้อมแสดงแผนที่
5. ตรวจว่าพิกัดอยู่ในพื้นที่จังหวัด
6. แยกเบอร์โทรหรือชื่อผู้ประสานงานเป็นข้อมูล internal
7. ใส่วันที่ปรับปรุงข้อมูล
8. export เป็น CSV UTF-8
9. import เข้าตารางเป้าหมาย
10. เปิดหน้า map หรือ dashboard เพื่อตรวจตำแหน่งจริง

## 18. โครงสร้างไฟล์ทำงานที่แนะนำ

ควรแยกไฟล์ในโฟลเดอร์ข้อมูลตามสถานะ เพื่อไม่ให้สับสนระหว่างไฟล์ต้นฉบับกับไฟล์พร้อมนำเข้า

```text
data/
  raw/
    production_large_plots_original_2569.xlsx
  working/
    production_large_plots_cleaning_2569_v01.xlsx
  ready_to_import/
    production_large_plots_2569_ready.csv
  imported/
    production_large_plots_2569_imported_2026-05-10.csv
  logs/
    production_large_plots_cleaning_log.md
```

แนวทางนี้ช่วยให้ย้อนกลับได้ว่าไฟล์ที่ import เข้า Dashboard มาจากไฟล์ใด และผ่านการแก้อะไรมาบ้าง

## 19. Checklist ก่อน import เข้า Supabase

- [ ] ไฟล์ต้นฉบับถูกสำรองแล้ว
- [ ] ไฟล์ clean ไม่มี merged cells
- [ ] แถวแรกเป็น header
- [ ] หนึ่งแถวคือหนึ่งรายการ
- [ ] ไม่มีแถวรวม แถวว่าง หรือหัวรายงานปนอยู่
- [ ] ชื่ออำเภอ/ตำบลสะกดตรงมาตรฐาน
- [ ] วันที่อยู่ในรูปแบบเดียวกัน
- [ ] ตัวเลขเป็นตัวเลขล้วน
- [ ] พิกัดอยู่ในรูปแบบ decimal degree ถ้ามี
- [ ] ตรวจรายการซ้ำแล้ว
- [ ] แยกข้อมูลส่วนบุคคลแล้ว
- [ ] ระบุระดับ public/internal/restricted แล้ว
- [ ] ตั้งชื่อคอลัมน์หรือเตรียม mapping แล้ว
- [ ] export เป็น CSV UTF-8 แล้ว
- [ ] เปิด CSV ตรวจภาษาไทยแล้วไม่เพี้ยน

## 20. หลัง import ต้องตรวจอะไรบ้าง

หลังนำข้อมูลเข้าระบบแล้ว ควรตรวจผลในระบบทันที

- [ ] จำนวนรายการในระบบตรงกับไฟล์ที่ import
- [ ] คอลัมน์สำคัญแสดงผลถูกต้อง
- [ ] ค้นหาด้วยชื่อ อำเภอ หรือ keyword แล้วพบข้อมูล
- [ ] Dashboard รวมตัวเลขถูกต้อง
- [ ] Map แสดงตำแหน่งถูกต้อง ถ้ามีพิกัด
- [ ] หน้า public ไม่แสดงข้อมูลที่ควรเป็น internal
- [ ] AI Chatbot ตอบข้อมูลจากชุดใหม่นี้ได้ถูกทิศทาง
- [ ] export จากระบบกลับมาแล้วข้อมูลไม่เพี้ยน

ถ้าพบปัญหา ควรบันทึกว่าเกิดจากไฟล์ต้นทาง การ clean การ map คอลัมน์ หรือโครงสร้างตาราง เพื่อแก้ให้ตรงจุดในรอบถัดไป

## 21. ผลลัพธ์ที่ควรได้จากขั้นตอนนี้

หลังจบบทนี้ ควรมีผลลัพธ์อย่างน้อย 5 อย่าง

1. ไฟล์ต้นฉบับที่เก็บแยกไว้
2. ไฟล์ทำงานที่มีบันทึกการแก้ไข
3. ไฟล์ CSV UTF-8 ที่พร้อมนำเข้า
4. ตาราง mapping ระหว่างคอลัมน์ไฟล์กับ field ในระบบ
5. Checklist ตรวจคุณภาพข้อมูลก่อน import

เมื่อเตรียมข้อมูลได้ครบแล้ว จึงเข้าสู่บทถัดไป คือการออกแบบฐานข้อมูลและตั้งค่า Supabase ให้รองรับข้อมูลแต่ละชุดอย่างปลอดภัยและใช้งานได้จริง
