# 02 การรวบรวมข้อมูลจากจังหวัด

ไฟล์นี้ใช้วางแนวทางการสำรวจ รวบรวม และขอข้อมูลจากกลุ่มงาน อำเภอ หรือแหล่งข้อมูลภายนอก ก่อนนำไปทำความสะอาดและนำเข้าระบบ `npt_dashboard`

บทนี้ยังไม่ลงรายละเอียดเรื่องการ clean Excel/CSV หรือการสร้างตาราง Supabase เพราะจะอยู่ในบทถัดไป จุดสำคัญของบทนี้คือทำให้รู้ว่า “ต้องเก็บข้อมูลอะไร จากใคร ใช้เพื่ออะไร และข้อมูลนั้นเปิดเผยได้แค่ไหน”

## 1. เป้าหมายของขั้นตอนนี้

การรวบรวมข้อมูลควรทำให้ได้ข้อมูลที่มีเจ้าของชัดเจน ใช้ต่อใน Dashboard ได้ และไม่เสี่ยงเรื่องสิทธิ์หรือข้อมูลส่วนบุคคล

เป้าหมายหลัก ได้แก่

- รู้ว่าจังหวัดมีชุดข้อมูลอะไรอยู่แล้ว
- รู้ว่าแต่ละชุดข้อมูลอยู่กับกลุ่มงานใดหรืออำเภอใด
- รู้ว่าข้อมูลชุดใดใช้กับหน้า public และชุดใดใช้เฉพาะภายใน
- รู้ว่าข้อมูลใดต้องปรับรูปแบบก่อนนำเข้าระบบ
- รู้ว่าข้อมูลใดต้องขออนุญาตก่อนเผยแพร่
- เตรียมตารางข้อมูลให้พร้อมสำหรับบทถัดไปเรื่องการทำความสะอาดข้อมูล

## 2. หลักคิดก่อนขอข้อมูล

ก่อนส่งหนังสือหรือแบบฟอร์มขอข้อมูล ควรตกลงขอบเขตให้ชัดก่อน ไม่เช่นนั้นข้อมูลที่ได้อาจซ้ำกัน ใช้ไม่ได้ หรือมีรายละเอียดเกินความจำเป็น

คำถามที่ควรถามก่อนเริ่ม ได้แก่

1. ข้อมูลนี้จะใช้ตอบคำถามอะไรใน Dashboard
2. ใครเป็นเจ้าของข้อมูลตัวจริง
3. ข้อมูลนี้ปรับปรุงล่าสุดเมื่อใด
4. ต้องอัปเดตบ่อยแค่ไหน
5. มีข้อมูลส่วนบุคคลหรือข้อมูลอ่อนไหวหรือไม่
6. เปิดเผยต่อประชาชนได้ทั้งหมด บางส่วน หรือไม่ได้เลย
7. มีรหัสอำเภอ ตำบล พิกัด หรือข้อมูลเชิงพื้นที่หรือไม่
8. มีไฟล์ต้นทางเป็น Excel, CSV, Google Sheet, ระบบกลาง หรือ API

## 3. กลุ่มข้อมูลหลักที่ควรรวบรวม

ระบบ `npt_dashboard` แบ่งข้อมูลตามกลุ่มงานหลัก เพื่อให้เจ้าหน้าที่รับผิดชอบและตรวจข้อมูลได้ง่าย

| กลุ่มงาน | ตัวอย่างข้อมูล | ตารางเป้าหมายในระบบ |
|---|---|---|
| บริหารทั่วไป | บุคลากร ทรัพย์สิน งบประมาณ | `personnel`, `assets`, `budgets` |
| ยุทธศาสตร์และสารสนเทศ | ทะเบียนเกษตรกร GIS พื้นที่เกษตร ศูนย์เรียนรู้ KPI ภัยพิบัติ สภาพอากาศ | `farmer_registry`, `gis_areas`, `agricultural_areas`, `learning_centers`, `kpi_plans`, `disasters`, `daily_weather` |
| ส่งเสริมและพัฒนาการผลิต | แปลงใหญ่ มาตรฐาน GAP ผลผลิตพืช | `large_plots`, `certifications`, `crop_production` |
| ส่งเสริมและพัฒนาเกษตรกร | วิสาหกิจชุมชน Smart Farmer กลุ่มเกษตรกร สถาบันเกษตรกร ท่องเที่ยวเชิงเกษตร | `community_enterprises`, `smart_farmers`, `farmer_groups`, `farmer_institutes`, `agri_tourism` |
| อารักขาพืช | แปลงพยากรณ์ ศจช. ศดปช. จุดความร้อน การระบาดศัตรูพืช ชีวภัณฑ์ | `forecast_plots`, `pest_centers`, `soil_fertilizer_centers`, `fire_hotspots`, `pest_outbreaks`, `biocontrol_stock` |
| คำขอข้อมูลจากอำเภอ | แบบขอข้อมูล รายการมอบหมาย คำตอบจากอำเภอ | `data_requests`, `data_request_assignments`, `data_request_responses` |

ตารางในช่องสุดท้ายเป็นชื่อที่ระบบใช้ในฐานข้อมูล ไม่จำเป็นต้องบังคับให้ไฟล์ Excel ต้นทางใช้ชื่อเดียวกัน แต่ควรทำ mapping ไว้ตั้งแต่ต้น

## 4. ทำบัญชีรายการข้อมูลก่อนขอข้อมูลจริง

ก่อนรวบรวมไฟล์ ควรทำ “บัญชีรายการข้อมูล” 1 ชุด เพื่อใช้คุมงานทั้งจังหวัด

ตัวอย่างคอลัมน์ที่ควรมีในบัญชีรายการข้อมูล:

| คอลัมน์ | คำอธิบาย |
|---|---|
| รหัสชุดข้อมูล | รหัสสั้น ๆ เช่น `STR-001`, `PROD-002` |
| ชื่อชุดข้อมูล | ชื่อที่เจ้าหน้าที่เข้าใจง่าย |
| กลุ่มงานเจ้าของข้อมูล | กลุ่มงานหรือฝ่ายที่รับผิดชอบ |
| ผู้ประสานงาน | ชื่อ/ตำแหน่ง/ช่องทางติดต่อ |
| แหล่งข้อมูล | Excel, Google Sheet, ระบบกลาง, API, รายงานเดิม |
| ปีข้อมูล | ปีงบประมาณหรือปี พ.ศ. ของข้อมูล |
| รอบการอัปเดต | รายวัน รายเดือน รายไตรมาส รายปี หรือตามเหตุการณ์ |
| ระดับการเผยแพร่ | public, internal, restricted |
| ตารางเป้าหมาย | ตาราง Supabase ที่จะนำเข้า |
| สถานะ | รอข้อมูล ได้รับแล้ว ตรวจแล้ว ต้องแก้ไข นำเข้าแล้ว |
| หมายเหตุ | ข้อจำกัด ปัญหา หรือเงื่อนไขการใช้ข้อมูล |

ควรเก็บบัญชีรายการนี้ไว้เป็นไฟล์กลาง เช่น Google Sheet หรือ Excel กลางของโครงการ เพื่อให้ทุกคนเห็นสถานะเดียวกัน

## 5. แยกระดับข้อมูลก่อนใช้งาน

การแบ่งระดับข้อมูลควรทำตั้งแต่ตอนรับข้อมูล ไม่ควรรอจนถึงตอนเผยแพร่หน้าเว็บ

| ระดับข้อมูล | ความหมาย | ตัวอย่างการใช้งาน |
|---|---|---|
| `public` | เปิดเผยต่อประชาชนได้ | จำนวนแปลงใหญ่ รายชื่อแหล่งท่องเที่ยวเกษตร ข้อมูลภาพรวมรายอำเภอ |
| `internal` | ใช้เฉพาะเจ้าหน้าที่ที่เข้าสู่ระบบ | รายละเอียดการปฏิบัติงาน รายงานภายใน ข้อมูลสำหรับติดตามงาน |
| `restricted` | จำกัดเฉพาะผู้มีหน้าที่เกี่ยวข้อง | ข้อมูลส่วนบุคคล ข้อมูลติดต่อ ข้อมูลที่ยังไม่ผ่านการอนุมัติ |

แนวปฏิบัติ:

- ถ้าไม่แน่ใจ ให้จัดเป็น `internal` หรือ `restricted` ไว้ก่อน
- ข้อมูลรายบุคคลควรสรุปเป็นภาพรวมก่อนนำไปแสดง public
- หน้า public ควรใช้ข้อมูลที่ตรวจแล้วและไม่กระทบสิทธิส่วนบุคคล
- ข้อมูลที่มีเลขบัตรประชาชน เบอร์โทร ที่อยู่ละเอียด หรือข้อมูลเฉพาะราย ต้องระวังเป็นพิเศษ

## 6. ข้อมูลขั้นต่ำที่ควรมีในแต่ละชุดข้อมูล

เพื่อให้ Dashboard, Search, Map และ AI Chatbot ใช้ข้อมูลได้ดี ควรวางมาตรฐานข้อมูลขั้นต่ำไว้ตั้งแต่ขั้นตอนขอข้อมูล

### 6.1 ข้อมูลระบุพื้นที่

ควรมีอย่างน้อยหนึ่งระดับ:

- จังหวัด
- อำเภอ
- ตำบล
- หมู่บ้าน
- พิกัดละติจูด/ลองจิจูด
- รหัสพื้นที่จากระบบราชการ ถ้ามี

สำหรับระบบนี้ควรใช้ชื่ออำเภอและตำบลให้สะกดเหมือนกันทุกไฟล์ เพราะมีผลต่อกราฟ แผนที่ และการรวมข้อมูล

### 6.2 ข้อมูลระบุเวลา

ควรระบุให้ชัดว่าเป็นข้อมูลของช่วงใด

- ปีงบประมาณ
- ปี พ.ศ.
- เดือน
- วันที่ปรับปรุงข้อมูลล่าสุด
- วันที่เกิดเหตุการณ์ เช่น วันที่สำรวจ วันที่รับรอง วันที่พบการระบาด

### 6.3 ข้อมูลตัวเลข

ตัวเลขควรแยกช่องให้ชัด ไม่รวมหน่วยไว้ในช่องเดียวกัน

ตัวอย่างที่ควรใช้:

| รายการ | รูปแบบที่ควรใช้ |
|---|---|
| พื้นที่ | ตัวเลขอย่างเดียว และมีคอลัมน์หน่วย เช่น ไร่ |
| จำนวนสมาชิก | ตัวเลขจำนวนเต็ม |
| งบประมาณ | ตัวเลข ไม่ใส่คำว่า บาทในช่องเดียวกัน |
| พิกัด | แยก `latitude` และ `longitude` |

### 6.4 ข้อมูลอ้างอิง

ควรมีข้อมูลที่ช่วยตรวจย้อนกลับ เช่น

- แหล่งที่มาของข้อมูล
- ผู้ให้ข้อมูล
- วันที่ได้รับข้อมูล
- ไฟล์ต้นทาง
- หมายเหตุการแก้ไข

ข้อมูลเหล่านี้ช่วยให้ตรวจสอบได้เมื่อ Dashboard แสดงผลผิดหรือมีตัวเลขไม่ตรงกับรายงานเดิม

## 7. วิธีรวบรวมข้อมูลที่แนะนำ

เลือกวิธีรวบรวมตามความพร้อมของหน่วยงานและลักษณะข้อมูล

| วิธี | เหมาะกับกรณี | ข้อควรระวัง |
|---|---|---|
| ขอไฟล์ Excel จากกลุ่มงาน | มีไฟล์เดิมอยู่แล้ว | ต้องตรวจหัวตาราง รูปแบบวันที่ และชื่ออำเภอ/ตำบล |
| ใช้ Google Sheet กลาง | ต้องการให้หลายอำเภอกรอกพร้อมกัน | ต้องกำหนดสิทธิ์แก้ไขและป้องกันการลบสูตร/หัวตาราง |
| ใช้ระบบ Data Request ใน Dashboard | ต้องการมอบหมายอำเภอและรวมคำตอบกลับเข้าระบบ | ต้องกำหนด schema คำถามให้ชัดก่อนเปิดรับข้อมูล |
| ดึงจาก API หรือระบบกลาง | ข้อมูลมีแหล่งกลางและอัปเดตบ่อย | ต้องตรวจเงื่อนไขการใช้งานและเสถียรภาพของ API |
| กรอกข้อมูลผ่านหน้า Data Management | ข้อมูลจำนวนไม่มากหรือแก้เฉพาะรายการ | ต้องกำหนดสิทธิ์ editor ให้ถูกกลุ่มงาน |

## 8. แนวทางใช้ระบบ Data Request

ระบบมีแนวคิดสำหรับให้จังหวัดสร้างคำขอข้อมูล แล้วมอบหมายให้อำเภอส่งข้อมูลกลับเข้าระบบ

โครงสร้างหลักมี 3 ส่วน:

| ตาราง | ใช้เก็บอะไร |
|---|---|
| `data_requests` | หัวข้อคำขอข้อมูล คำอธิบาย schema ของแบบฟอร์ม ลิงก์ Google Sheet สถานะ และกำหนดส่ง |
| `data_request_assignments` | รายการมอบหมายว่าอำเภอใดต้องส่งข้อมูลในคำขอนั้น และส่งแล้วหรือยัง |
| `data_request_responses` | คำตอบที่แต่ละอำเภอส่งกลับมาในรูปแบบ JSON |

สถานะคำขอข้อมูลควรใช้ให้ชัด:

| สถานะ | ความหมาย |
|---|---|
| `draft` | ยังเป็นร่าง ใช้เตรียมแบบฟอร์ม |
| `published` | เปิดรับข้อมูลจากอำเภอแล้ว |
| `closed` | ปิดรับข้อมูลแล้ว ใช้สำหรับตรวจและนำไปใช้ต่อ |

ขั้นตอนแนะนำ:

1. จังหวัดกำหนดหัวข้อคำขอข้อมูล
2. ระบุคำอธิบายและกำหนดส่ง
3. กำหนดช่องข้อมูลที่ต้องการ เช่น อำเภอ ตำบล รายการ จำนวน พื้นที่ หมายเหตุ
4. เลือกอำเภอที่ต้องส่งข้อมูล
5. เปิดสถานะเป็น `published`
6. อำเภอกรอกข้อมูลหรือแนบข้อมูลตามรูปแบบที่กำหนด
7. จังหวัดตรวจความครบถ้วนและ export ผลลัพธ์
8. นำข้อมูลที่ผ่านการตรวจแล้วไป clean และ map เข้าตารางจริงของระบบ

## 9. ตัวอย่างแบบฟอร์มขอข้อมูลจากอำเภอ

ตัวอย่างหัวข้อ: ขอข้อมูลจุดเรียนรู้ด้านการเกษตรระดับอำเภอ

| ช่องข้อมูล | ชนิดข้อมูล | จำเป็น | หมายเหตุ |
|---|---|---|---|
| อำเภอ | select | ใช่ | เลือกจากรายชื่ออำเภอ |
| ตำบล | text | ใช่ | ใช้ชื่อทางราชการ |
| ชื่อจุดเรียนรู้ | text | ใช่ | ชื่อสถานที่หรือกลุ่ม |
| ประเภทจุดเรียนรู้ | select | ใช่ | เช่น ศพก., ศูนย์เครือข่าย, แหล่งเรียนรู้อื่น |
| สินค้าหรือกิจกรรมเด่น | text | ไม่บังคับ | ใช้แสดงในหน้า public ได้ถ้าผ่านการตรวจ |
| พื้นที่โดยประมาณ | number | ไม่บังคับ | หน่วยเป็นไร่ |
| latitude | number | ไม่บังคับ | ใช้กับแผนที่ |
| longitude | number | ไม่บังคับ | ใช้กับแผนที่ |
| ผู้ประสานงาน | text | ไม่บังคับ | ใช้ภายในเท่านั้น |
| หมายเหตุ | textarea | ไม่บังคับ | ใช้สำหรับอธิบายเพิ่มเติม |

หลังรับข้อมูล ควรตรวจว่า field ใดจะเข้าหน้า public และ field ใดต้องใช้เฉพาะภายใน

## 10. การตั้งชื่อไฟล์ข้อมูล

ควรตั้งชื่อไฟล์ให้รู้ทันทีว่าเป็นข้อมูลอะไร ของปีไหน และมาจากใคร

รูปแบบที่แนะนำ:

```text
[กลุ่มงาน]_[ชื่อข้อมูล]_[พื้นที่]_[ปีข้อมูล]_[วันที่ส่ง].xlsx
```

## 11. การตรวจรับข้อมูลเบื้องต้น

เมื่อได้รับข้อมูลแล้ว ควรตรวจทันที ไม่ควรรอจนถึงขั้นตอนนำเข้า Supabase

Checklist ตรวจรับข้อมูล:

- [ ] เปิดไฟล์ได้และไม่มีรหัสผ่านที่ทีมไม่ทราบ
- [ ] มีหัวตารางชัดเจน
- [ ] ไม่มีการรวมเซลล์ในส่วนข้อมูลหลัก
- [ ] หนึ่งแถวแทนหนึ่งรายการข้อมูล
- [ ] ไม่มีแถวสรุปรวมปนอยู่กับข้อมูลหลัก
- [ ] ชื่ออำเภอและตำบลสะกดสม่ำเสมอ
- [ ] วันที่อยู่ในรูปแบบที่อ่านได้
- [ ] ตัวเลขไม่ปนหน่วยในช่องเดียวกัน
- [ ] พิกัดอยู่ในรูปแบบละติจูด/ลองจิจูด ถ้ามี
- [ ] ข้อมูลส่วนบุคคลถูกแยกหรือทำเครื่องหมายไว้แล้ว
- [ ] ระบุแหล่งที่มาและวันที่ปรับปรุงข้อมูลแล้ว

## 12. Mapping ข้อมูลกับโมดูลในระบบ

หลังรวบรวมข้อมูล ควรระบุให้ได้ว่าข้อมูลแต่ละชุดจะไปแสดงในส่วนใดของระบบ

| ข้อมูล | ใช้กับส่วนใดได้บ้าง |
|---|---|
| พื้นที่เกษตรรายอำเภอ | Dashboard, Interactive Dashboard, AI Chatbot, Search |
| แปลงใหญ่ | Public detail page, Dashboard, Map, Search, AI Chatbot |
| วิสาหกิจชุมชน | Public detail page, Dashboard, Search, AI Chatbot |
| Smart Farmer | Public summary หรือ internal table ขึ้นกับระดับข้อมูล |
| ศูนย์เรียนรู้ | Public page, Map, Dashboard |
| จุดความร้อน | Protection dashboard, Map, Widget, AI Chatbot |
| งบประมาณ | Internal dashboard, Admin module, Search ภายใน |
| คำขอข้อมูลจากอำเภอ | Data Request, export ผลลัพธ์, นำไป clean ต่อ |

ถ้าข้อมูลชุดใดใช้กับหลายหน้า ควรกำหนด field ให้ครบตั้งแต่ต้น เพราะการกลับไปขอข้อมูลเพิ่มภายหลังมักใช้เวลามาก

## 13. ข้อควรระวังเรื่องข้อมูลราชการ

ข้อมูลราชการบางชุดนำเสนอเป็นภาพรวมได้ แต่ไม่ควรเผยแพร่รายละเอียดรายบุคคลหรือรายละเอียดที่ทำให้ระบุตัวบุคคลได้ง่าย

ข้อควรระวัง:

- หลีกเลี่ยงการเผยแพร่เลขบัตรประชาชน เบอร์โทร ที่อยู่ละเอียด หรือข้อมูลติดต่อส่วนตัว
- ถ้าเป็นรายชื่อบุคคลหรือเกษตรกร ให้ตรวจสิทธิ์และวัตถุประสงค์ก่อนใช้
- ถ้าเป็นข้อมูลจากระบบกลาง ให้ตรวจเงื่อนไขการนำข้อมูลออกมาใช้
- ข้อมูลที่ยังไม่ตรวจหรือยังไม่อนุมัติ ไม่ควรแสดงในหน้า public
- ควรเก็บไฟล์ต้นทางไว้ในพื้นที่ที่จำกัดสิทธิ์ ไม่วางไว้ใน public repository

## 14. บทบาทผู้เกี่ยวข้องในการรวบรวมข้อมูล

| บทบาท | หน้าที่ |
|---|---|
| เจ้าของโครงการ | กำหนดขอบเขตและอนุมัติชุดข้อมูลหลัก |
| ผู้ดูแลระบบ | จัดโครงสร้างไฟล์ ตรวจสิทธิ์ และเตรียมช่องทางนำเข้า |
| กลุ่มงานเจ้าของข้อมูล | ส่งข้อมูล ตรวจความถูกต้อง และอธิบายความหมายของข้อมูล |
| อำเภอ | กรอกหรือยืนยันข้อมูลในพื้นที่ |
| ผู้ตรวจข้อมูล | ตรวจความครบถ้วน ความซ้ำ และรูปแบบข้อมูล |
| Admin ระบบ | จัดการสิทธิ์ ตาราง และการเชื่อมกับ Dashboard |

ควรกำหนดผู้รับผิดชอบของแต่ละชุดข้อมูลให้ชัด เพราะเมื่อมีตัวเลขผิดหรือข้อมูลไม่อัปเดต จะได้รู้ว่าต้องประสานใคร

## 15. Checklist ก่อนส่งข้อมูลไปทำความสะอาด

- [ ] มีบัญชีรายการข้อมูลกลางแล้ว
- [ ] ระบุเจ้าของข้อมูลครบทุกชุดแล้ว
- [ ] ระบุระดับการเผยแพร่ของแต่ละชุดแล้ว
- [ ] ระบุไฟล์ต้นทางหรือแหล่งข้อมูลแล้ว
- [ ] ตรวจรูปแบบไฟล์เบื้องต้นแล้ว
- [ ] ตรวจชื่ออำเภอ ตำบล ปีข้อมูล และวันที่แล้ว
- [ ] แยกข้อมูล public/internal/restricted แล้ว
- [ ] ระบุตารางเป้าหมายในระบบแล้ว
- [ ] ระบุรอบการอัปเดตข้อมูลแล้ว
- [ ] เก็บไฟล์ต้นทางไว้ในพื้นที่ที่เหมาะสมแล้ว

## 16. ผลลัพธ์ที่ควรได้จากขั้นตอนนี้

หลังจบบทนี้ ควรมีเอกสารหรือไฟล์กลางอย่างน้อย 4 อย่าง

1. บัญชีรายการข้อมูลของจังหวัด
2. รายชื่อเจ้าของข้อมูลและผู้ประสานงานแต่ละชุด
3. ไฟล์ข้อมูลต้นทางที่รวบรวมแล้ว
4. ตาราง mapping ว่าข้อมูลแต่ละชุดจะเข้าตารางใดและใช้แสดงในหน้าใดของระบบ

เมื่อได้ผลลัพธ์เหล่านี้แล้ว จึงเข้าสู่บทถัดไป คือการทำความสะอาดข้อมูลและเตรียมไฟล์ให้พร้อมสำหรับนำเข้า Supabase
