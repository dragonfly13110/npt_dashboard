# 08 การดูแลระบบและอบรมผู้ใช้งาน

ไฟล์นี้เป็นคู่มือการดูแลระบบ `npt_dashboard` หลังเปิดใช้งานจริง รวมถึงแนวทางอบรมผู้ใช้งาน การกำหนดรอบอัปเดตข้อมูล การรับแจ้งปัญหา การตรวจ audit log การสำรองข้อมูล และการวางแผนพัฒนาระบบต่อเนื่อง

บทนี้ต่อจากบท 07 ที่อธิบายเรื่องความปลอดภัยและการ deploy แล้ว เมื่อระบบถูก deploy ขึ้นใช้งานจริง งานสำคัญจะเปลี่ยนจาก “สร้างระบบให้เปิดได้” เป็น “ทำให้ระบบมีข้อมูลถูกต้อง ใช้งานได้ต่อเนื่อง และเจ้าหน้าที่ใช้เป็นจริง”

## 1. เป้าหมายของบทนี้

เมื่อทำตามบทนี้จบ ควรทำได้ดังนี้

- กำหนดผู้รับผิดชอบระบบหลังเปิดใช้งานได้
- วางรอบอัปเดตข้อมูลของแต่ละกลุ่มงานได้
- จัดการผู้ใช้ role และ department ได้เป็นระบบ
- ตรวจข้อมูลผิดปกติและแก้ไขข้อมูลได้
- ตรวจ audit log และประวัติการแก้ไขได้
- วางรอบ backup ข้อมูลและไฟล์สำคัญได้
- สร้างช่องทางรับแจ้งปัญหาและติดตามสถานะได้
- วางแผนอบรมผู้ใช้แต่ละกลุ่มได้
- ทำคู่มือย่อยหรือสื่อสอนผู้ใช้งานได้
- วางแผนปรับปรุงระบบเป็นรอบ ๆ โดยไม่กระทบ production

## 2. หลักคิดหลังเปิดใช้งานจริง

ระบบ Dashboard จะมีประโยชน์ต่อเมื่อข้อมูลถูกต้องและผู้ใช้เชื่อถือระบบ

สิ่งที่ต้องดูแลหลังเปิดใช้งานมี 5 เรื่องหลัก

| เรื่อง | เป้าหมาย |
|---|---|
| ข้อมูล | ถูกต้อง ทันเวลา ตรวจย้อนกลับได้ |
| ผู้ใช้ | เข้าใช้งานได้ตามสิทธิ์ ไม่ให้สิทธิ์เกินจำเป็น |
| ระบบ | เปิดได้ เสถียร มีแผน backup และ rollback |
| ความปลอดภัย | ไม่เปิดข้อมูลเกินสิทธิ์ ไม่ทำ key หลุด |
| การใช้งาน | เจ้าหน้าที่รู้วิธีใช้และรู้ว่าต้องแจ้งปัญหาที่ไหน |

ถ้าระบบเปิดได้แต่ไม่มีคนอัปเดตข้อมูล ข้อมูลจะเริ่มเก่าและผู้ใช้จะเลิกเชื่อถือระบบ

ถ้าข้อมูลถูกต้องแต่ผู้ใช้ไม่รู้วิธีค้นหา สรุป หรือ export ระบบก็จะไม่ช่วยลดงานได้เท่าที่ควร

## 3. บทบาทผู้ดูแลระบบ

ควรกำหนดผู้รับผิดชอบอย่างน้อย 4 บทบาท

| บทบาท | หน้าที่หลัก | ผู้ที่เหมาะสม |
|---|---|---|
| System Admin | ดูแลผู้ใช้ สิทธิ์ deploy และแก้ปัญหาเทคนิค | เจ้าหน้าที่ที่ดูแลระบบหรือทีมพัฒนา |
| Data Owner | รับผิดชอบความถูกต้องของข้อมูลแต่ละชุด | กลุ่มงานเจ้าของข้อมูล |
| Data Editor | เพิ่ม แก้ไข import และตรวจข้อมูล | เจ้าหน้าที่กลุ่มงานหรืออำเภอ |
| Viewer / Executive User | ดู dashboard และนำข้อมูลไปใช้ตัดสินใจ | ผู้บริหารหรือเจ้าหน้าที่ทั่วไป |

ควรกำหนดชื่อผู้รับผิดชอบจริงไว้ในเอกสารกลาง เช่น

```text
ระบบ: npt_dashboard
System Admin: ชื่อ / เบอร์ภายใน / email
Data Owner กลุ่มยุทธศาสตร์ฯ: ชื่อ / email
Data Owner กลุ่มส่งเสริมการผลิต: ชื่อ / email
Data Owner กลุ่มพัฒนาเกษตรกร: ชื่อ / email
Data Owner กลุ่มอารักขาพืช: ชื่อ / email
Data Owner ฝ่ายบริหารทั่วไป: ชื่อ / email
```

## 5. รอบการอัปเดตข้อมูล

ข้อมูลแต่ละชุดไม่จำเป็นต้องอัปเดตถี่เท่ากัน ควรกำหนดรอบให้เหมาะกับลักษณะข้อมูล

| ประเภทข้อมูล | รอบอัปเดตที่แนะนำ | ตัวอย่างตาราง |
|---|---|---|
| ข้อมูลค่อนข้างคงที่ | รายปี หรือเมื่อมีการเปลี่ยนแปลง | `personnel`, `assets`, `learning_centers` |
| ข้อมูลแผนงาน/งบประมาณ | รายเดือน หรือรายไตรมาส | `budgets`, `kpi_plans` |
| ข้อมูลส่งเสริมการผลิต | รายไตรมาส หรือรายปี | `large_plots`, `certifications`, `crop_production` |
| ข้อมูลกลุ่มเกษตรกร | รายไตรมาส หรือรายปี | `community_enterprises`, `smart_farmers`, `farmer_groups` |
| ข้อมูลอารักขาพืช | รายเดือน หรือตามเหตุการณ์ | `pest_outbreaks`, `forecast_plots`, `fire_hotspots` |
| ข้อมูลสภาพอากาศ/รายวัน | รายวันหรืออัตโนมัติ | `daily_weather` |
| คำขอข้อมูลจากอำเภอ | ตามรอบมอบหมายงาน | `data_requests`, `data_request_responses` |

ตัวอย่างตารางควบคุมรอบข้อมูล

| ชุดข้อมูล | เจ้าของข้อมูล | รอบอัปเดต | วันครบกำหนด | วิธีส่งข้อมูล | สถานะล่าสุด |
|---|---|---|---|---|---|
| แปลงใหญ่ | กลุ่มส่งเสริมการผลิต | รายไตรมาส | ทุกวันที่ 5 หลังจบไตรมาส | Import CSV | รอตรวจ |
| วิสาหกิจชุมชน | กลุ่มพัฒนาเกษตรกร | รายไตรมาส | ทุกวันที่ 10 หลังจบไตรมาส | Data Management | อัปเดตแล้ว |
| จุดความร้อน | กลุ่มอารักขาพืช | รายเดือน | วันที่ 3 ของเดือนถัดไป | API/CSV | รอข้อมูล |

## 8. การตรวจคุณภาพข้อมูลประจำรอบ

ทุกเดือนหรือทุกไตรมาสควรตรวจคุณภาพข้อมูล

### 8.1 ตรวจชื่ออำเภอ

ใช้ SQL เพื่อตรวจค่าที่ไม่ตรงมาตรฐาน

```sql
SELECT DISTINCT district
FROM large_plots
ORDER BY district;
```

รายชื่อที่ควรพบสำหรับนครปฐม

```text
เมืองนครปฐม
กำแพงแสน
นครชัยศรี
ดอนตูม
บางเลน
สามพราน
พุทธมณฑล
```

ถ้าพบค่าเช่น `เมือง`, `อ.เมืองนครปฐม`, `นครชัยศริ` ให้แก้ให้ตรงมาตรฐาน

### 8.2 ตรวจค่าว่างใน field สำคัญ

ตัวอย่าง

```sql
SELECT COUNT(*)
FROM large_plots
WHERE district IS NULL OR district = '';
```

```sql
SELECT COUNT(*)
FROM community_enterprises
WHERE enterprise_name IS NULL OR enterprise_name = '';
```

### 8.3 ตรวจตัวเลขผิดปกติ

ตัวอย่าง

```sql
SELECT *
FROM large_plots
WHERE area_rai < 0 OR member_count < 0;
```

```sql
SELECT *
FROM budgets
WHERE spent_amount > budget_amount;
```

### 8.4 ตรวจพิกัดผิดจังหวัด

สำหรับนครปฐม พิกัดโดยประมาณควรอยู่แถว latitude 13.x และ longitude 99.x ถึง 100.x

```sql
SELECT *
FROM agri_tourism
WHERE latitude IS NOT NULL
AND longitude IS NOT NULL
AND (
  latitude < 13 OR latitude > 14.5
  OR longitude < 99 OR longitude > 101
);
```

### 8.5 ตรวจข้อมูลซ้ำ

ตัวอย่างแปลงใหญ่

```sql
SELECT plot_name, district, COUNT(*)
FROM large_plots
GROUP BY plot_name, district
HAVING COUNT(*) > 1;
```

ตัวอย่างวิสาหกิจชุมชน

```sql
SELECT enterprise_name, district, COUNT(*)
FROM community_enterprises
GROUP BY enterprise_name, district
HAVING COUNT(*) > 1;
```

## 9. การจัดการผู้ใช้

การจัดการผู้ใช้ควรทำเป็นรอบและมีผู้อนุมัติ

### 9.1 เพิ่มผู้ใช้ใหม่

ขั้นตอน

1. ตรวจว่าผู้ใช้เป็นเจ้าหน้าที่หรือผู้เกี่ยวข้องจริง
2. สร้าง user ใน Supabase Auth หรือให้สมัครตาม flow ที่กำหนด
3. ตรวจว่ามี row ใน `profiles`
4. ตั้ง `full_name`, `department`, `role`
5. ให้ผู้ใช้ login ทดสอบ
6. ตรวจว่าเห็นเมนูตามสิทธิ์
7. บันทึกในทะเบียนผู้ใช้

ตัวอย่าง SQL

```sql
UPDATE profiles
SET
  full_name = 'ชื่อผู้ใช้',
  department = 'กลุ่มส่งเสริมและพัฒนาการผลิต',
  role = 'editor',
  updated_at = NOW()
WHERE email = 'user@example.com';
```

### 9.2 ปรับสิทธิ์ผู้ใช้

ควรปรับสิทธิ์เมื่อ

- ย้ายกลุ่มงาน
- เปลี่ยนหน้าที่
- หมดภารกิจในระบบ
- มีความเสี่ยงด้านข้อมูล
- ผู้ใช้ลาออกหรือย้ายหน่วยงาน

ตัวอย่างลดสิทธิ์เป็น viewer

```sql
UPDATE profiles
SET role = 'viewer', updated_at = NOW()
WHERE email = 'user@example.com';
```

### 9.3 ปิดสิทธิ์ผู้ใช้

ถ้าผู้ใช้ไม่ควรเข้าใช้งานแล้ว ให้ดำเนินการใน Supabase Auth

แนวทาง

- ลบ user หรือ disable ตามนโยบายที่ใช้
- ปรับ role ใน `profiles` ไม่ให้เป็น admin/editor
- ตรวจ audit log ว่ามีการใช้งานผิดปกติหรือไม่
- บันทึกวันที่ปิดสิทธิ์

## 10. รอบตรวจผู้ใช้ประจำเดือน

ควรตรวจผู้ใช้เดือนละ 1 ครั้ง

SQL ตรวจรายชื่อ

```sql
SELECT email, full_name, department, role, updated_at
FROM profiles
ORDER BY role, department, email;
```

Checklist

- [ ] ไม่มี admin เกินจำเป็น
- [ ] ไม่มี user ทดสอบค้างอยู่ใน production
- [ ] editor อยู่ถูกกลุ่มงานหรืออำเภอ
- [ ] ผู้ใช้ที่ย้ายงานถูกปรับสิทธิ์แล้ว
- [ ] ผู้ใช้ที่ไม่ได้ใช้งานนานถูกตรวจสอบแล้ว
- [ ] ไม่มี email แปลกหรือไม่รู้ที่มา

## 11. การตรวจ Audit Log

Audit log ใช้ตรวจว่าใครเพิ่ม แก้ไข หรือลบข้อมูลใด

ควรตรวจเมื่อ

- ตัวเลขใน dashboard ผิดปกติ
- ข้อมูลหาย
- มีการแก้ข้อมูลจำนวนมาก
- ผู้ใช้แจ้งว่าข้อมูลเปลี่ยนเอง
- ก่อนและหลัง import ข้อมูลชุดใหญ่

SQL ตัวอย่าง

```sql
SELECT user_email, action, table_name, record_id, created_at
FROM audit_logs
ORDER BY created_at DESC
LIMIT 100;
```

ตรวจเฉพาะตาราง

```sql
SELECT *
FROM audit_logs
WHERE table_name = 'large_plots'
ORDER BY created_at DESC
LIMIT 50;
```

ตรวจเฉพาะผู้ใช้

```sql
SELECT *
FROM audit_logs
WHERE user_email = 'user@example.com'
ORDER BY created_at DESC;
```

ข้อควรระวัง

- audit log อาจมีข้อมูลส่วนบุคคลใน `old_data` และ `new_data`
- ควรให้เฉพาะ admin หรือผู้ตรวจสอบที่ได้รับอนุญาตดูได้
- ถ้าต้องส่งออก audit log ให้ตรวจ field sensitive ก่อน

## 14. การกู้คืนข้อมูลเบื้องต้น

ถ้าข้อมูลผิดจากการ import หรือแก้ไข ให้ทำตามลำดับ

1. หยุดแก้ไขข้อมูลชุดนั้นก่อน
2. ตรวจ audit log ว่าเกิดอะไรขึ้น
3. ตรวจ backup ล่าสุด
4. ตรวจว่าแก้เฉพาะบาง record ได้หรือจำเป็นต้อง restore ทั้งตาราง
5. ถ้าเป็นตารางใหญ่ ให้ทดสอบ restore ใน staging ก่อน
6. แจ้งผู้เกี่ยวข้องก่อนแก้ข้อมูล production
7. หลัง restore ให้ตรวจ Dashboard/Search/AI อีกครั้ง

ตัวอย่างกรณีแก้เฉพาะรายการผิด

```sql
UPDATE large_plots
SET area_rai = 120.5, updated_at = NOW()
WHERE id = 'record-id-here';
```

ตัวอย่างกรณีลบข้อมูลที่ import ผิดรอบ

ถ้ามี field `source_file` หรือ `data_period` จะจัดการได้ง่ายกว่า

```sql
DELETE FROM large_plots
WHERE source_file = 'large_plots_2569_q2_wrong.csv';
```

ถ้ายังไม่มี field เหล่านี้ ควรเพิ่มในอนาคตเพื่อช่วย rollback ข้อมูล

## 16. การจัดระดับความรุนแรงของปัญหา

ใช้ระดับความรุนแรงเพื่อจัดลำดับงาน

| ระดับ | ความหมาย | ตัวอย่าง | แนวทางจัดการ |
|---|---|---|---|
| Critical | ระบบหลักใช้งานไม่ได้ | login ไม่ได้ทั้งระบบ, dashboard ล่ม | แก้ทันที หรือ rollback |
| High | กระทบงานสำคัญ | import ข้อมูลไม่ได้, ข้อมูลหายบางตาราง | เร่งตรวจและแจ้งผู้เกี่ยวข้อง |
| Medium | กระทบบางหน้า | chart ไม่แสดง, search บางคำไม่เจอ | เข้าคิวแก้ในรอบถัดไป |
| Low | ข้อความหรือ UX | คำสะกดผิด, ปุ่มจัดวางไม่สวย | รวมแก้พร้อมงานปรับปรุง |
| Request | ขอเพิ่มความสามารถ | เพิ่มกราฟ, เพิ่ม export, เพิ่ม field | ประเมินและจัดลำดับ |

- เปิดเมนู Data Request
- ดูคำขอข้อมูลที่ได้รับมอบหมาย
- อ่านคำอธิบายและกำหนดส่ง

- แก้ไขคำตอบก่อนปิดรอบ
- แจ้งปัญหาถ้าแบบฟอร์มไม่ตรงกับข้อมูลจริง

ตัวอย่างโจทย์ฝึก

```text
1. เปิดคำขอข้อมูลตัวอย่าง
2. กรอกข้อมูล 1 รายการ
3. บันทึกคำตอบ
4. ตรวจสถานะว่าส่งแล้ว
5. แก้ไขคำตอบก่อนปิดรอบ
```

## 30. ตัวอย่าง FAQ สำหรับผู้ใช้งาน

### 30.1 Login ไม่ได้ต้องทำอย่างไร

ตรวจ email/password ก่อน ถ้ายังเข้าไม่ได้ให้แจ้ง admin พร้อม screenshot หน้า error

ข้อมูลที่ต้องส่งให้ admin

```text
email ที่ใช้ login
เวลาที่พบปัญหา
ข้อความ error
รูปหน้าจอ
```

### 30.2 ข้อมูลไม่ขึ้นในหน้า Dashboard

สาเหตุที่เป็นไปได้

- ยังไม่ได้ import ข้อมูล
- RLS ไม่อนุญาตให้ role นี้อ่านข้อมูล
- ข้อมูลอยู่คนละ Supabase project
- ชื่ออำเภอหรือ field สำคัญไม่ตรง
- Dashboard ยัง cache ข้อมูลเก่า

แนวทางเบื้องต้น

- กด refresh
- ตรวจหน้าตารางข้อมูลว่ามีข้อมูลจริงหรือไม่
- แจ้ง admin พร้อมชื่อหน้าและคำอธิบาย

### 30.3 Import CSV แล้ว error

ตรวจไฟล์ CSV

- header อยู่แถวแรกหรือไม่
- encoding เป็น UTF-8 หรือไม่
- คอลัมน์วันที่และตัวเลขอยู่ในรูปแบบที่ถูกต้องหรือไม่
- มี merged cell หรือแถวว่างหรือไม่
- map คอลัมน์ถูก field หรือไม่

### 30.4 Search ไม่เจอข้อมูล

ตรวจ

- คำค้นสั้นเกินไปหรือไม่
- ข้อมูลนั้นอยู่ในตารางจริงหรือไม่
- ค้นจาก field ที่ระบบตั้งไว้หรือไม่
- ผู้ใช้มีสิทธิ์เห็นข้อมูลนั้นหรือไม่

### 30.5 AI ตอบไม่ตรงข้อมูล

ให้ตรวจ

- ข้อมูลใน Supabase ถูกต้องหรือไม่
- คำถามระบุอำเภอหรือช่วงข้อมูลชัดหรือไม่
- AI ใช้ข้อมูลตัวอย่างแทน aggregated stats หรือไม่
- ตารางนั้นถูกเพิ่มใน config ของ AI แล้วหรือไม่

## 31. รอบประชุมติดตามระบบ

หลังเปิดใช้งาน ควรมีรอบประชุมสั้น ๆ

| รอบ | วัตถุประสงค์ |
|---|---|
| สัปดาห์แรกหลังเปิด | เก็บปัญหาเร่งด่วน |
| สิ้นเดือนแรก | ตรวจ adoption และข้อมูลผิดพลาด |
| รายไตรมาส | ปรับระบบตามงานจริง |
| รายปี | ทบทวนชุดข้อมูลทั้งหมดและแผนพัฒนาระบบ |

หัวข้อประชุมรายเดือน

- ข้อมูลชุดใดอัปเดตแล้ว
- ข้อมูลชุดใดล่าช้า
- ผู้ใช้เจอปัญหาอะไรบ่อย
- Dashboard ส่วนใดมีคนใช้มาก/น้อย
- Search/AI ช่วยลดงานได้จริงหรือไม่
- มีข้อมูลใดควรเพิ่มหรือลด
- มีความเสี่ยงด้านข้อมูลส่วนบุคคลหรือไม่

## 32. ตัวชี้วัดการใช้งานระบบ

ควรมีตัวชี้วัดง่าย ๆ เพื่อติดตามว่าระบบมีประโยชน์จริงหรือไม่

| ตัวชี้วัด | วิธีดู |
|---|---|
| จำนวนผู้ใช้ที่ login รายเดือน | Supabase Auth หรือ log เพิ่มเติม |
| จำนวนครั้งเข้าหน้า Dashboard | `site_statistics` หรือ analytics |
| จำนวนครั้งค้นหา | เพิ่ม log ใน Search ในอนาคต |
| จำนวนคำถาม Chatbot | log เฉพาะ metadata ในอนาคต |
| จำนวนข้อมูลที่อัปเดตตามรอบ | changelog ข้อมูล |
| จำนวนปัญหาที่ผู้ใช้แจ้ง | issue tracker / Google Form |
| เวลาเฉลี่ยในการแก้ปัญหา | issue tracker |
| จำนวนรายงานที่ลดการทำมือ | ประเมินจากกลุ่มงาน |

## 33. การวางแผนพัฒนาระบบต่อ

ควรแยกงานพัฒนาเป็น 3 ประเภท

| ประเภท | ตัวอย่าง | วิธีจัดการ |
|---|---|---|
| Bug | login ไม่ได้, import error, กราฟไม่แสดง | แก้ตามความรุนแรง |
| Improvement | ปรับ UI, เพิ่ม filter, เพิ่ม export | รวมเป็นรอบ release |
| Feature | เพิ่มโมดูลใหม่, เพิ่ม API, เพิ่มรายงาน | วิเคราะห์ข้อมูลและสิทธิ์ก่อนทำ |

ก่อนเพิ่ม feature ใหม่ ให้ถาม 6 ข้อ

1. ใช้ตอบคำถามงานอะไร
2. ใครเป็นเจ้าของข้อมูล
3. ข้อมูลอัปเดตบ่อยแค่ไหน
4. มีข้อมูลส่วนบุคคลหรือไม่
5. ต้องแสดง public หรือ internal
6. ต้องให้ Dashboard, Search และ AI ใช้ข้อมูลนี้ด้วยหรือไม่

## 34. Release Workflow ที่แนะนำ

```text
รับ requirement
  ↓
ประเมินข้อมูลและสิทธิ์
  ↓
สร้าง branch ใหม่
  ↓
พัฒนาใน local
  ↓
ทดสอบกับข้อมูลจำลอง
  ↓
preview deploy หรือ staging
  ↓
ให้เจ้าของงานตรวจรับ
  ↓
merge เข้า main
  ↓
deploy production
  ↓
บันทึก changelog
```

ไม่ควรแก้ feature ใหญ่บน `main` โดยตรง

## 35. Release Note

ทุกครั้งที่มีการแก้ระบบสำคัญ ควรเขียน release note สั้น ๆ

ตัวอย่าง

```text
Release: 2026-05-12
ประเภท: Improvement
ผู้ deploy: admin@example.com

รายการเปลี่ยนแปลง:
1. เพิ่มกราฟแปลงใหญ่รายอำเภอ
2. แก้การสะกดอำเภอนครชัยศรีใน Dashboard config
3. เพิ่ม field ค้นหาในวิสาหกิจชุมชน

ผลกระทบ:
- ไม่มีการเปลี่ยน schema
- ผู้ใช้ไม่ต้องทำอะไรเพิ่มเติม

ตรวจหลัง deploy:
- Dashboard เปิดได้
- Search แปลงใหญ่ทำงาน
- Chatbot ตอบข้อมูลแปลงใหญ่ได้
```

## 38. สิ่งที่ไม่ควรทำหลังเปิดใช้งานจริง

- ไม่ควรให้ทุกคนเป็น admin
- ไม่ควรแก้ production database โดยไม่มี backup
- ไม่ควร import ข้อมูลจริงโดยไม่ตรวจไฟล์ก่อน
- ไม่ควรลบข้อมูลจำนวนมากโดยไม่บันทึกสาเหตุ
- ไม่ควรใส่ service role key ใน frontend
- ไม่ควรให้ AI อ่านข้อมูลส่วนบุคคลเต็มชุดโดยไม่จำเป็น
- ไม่ควรเพิ่มตารางใหม่โดยไม่คิดเรื่อง Search, AI, Dashboard และ RLS
- ไม่ควร deploy ก่อนทดสอบ build
- ไม่ควรใช้ production เป็นที่ทดลองโค้ดใหม่

## 40. ผลลัพธ์ที่ควรได้จากบทนี้

หลังจบบทนี้ ควรได้ผลลัพธ์ดังนี้

- มีรอบอัปเดตข้อมูลที่ตกลงร่วมกันแล้ว
- มี checklist ก่อนและหลัง import ข้อมูล
