# 05 การติดตั้งและตั้งค่าโปรเจกต์

ไฟล์นี้เป็นคู่มือการติดตั้งและตั้งค่าโปรเจกต์ `npt_dashboard` บนเครื่องพัฒนา เพื่อให้เจ้าหน้าที่หรือทีมพัฒนาสามารถ clone repo, ติดตั้ง dependencies, ตั้งค่า environment variables, เชื่อม Supabase, รันระบบบนเครื่อง, ทดสอบ build และตรวจว่าโมดูลหลักทำงานได้ก่อน deploy จริง

บทนี้ต่อจากบท 04 ที่พูดเรื่องการออกแบบฐานข้อมูลและตั้งค่า Supabase แล้ว ขั้นตอนนี้คือการนำ frontend ของระบบมาเชื่อมกับฐานข้อมูลที่เตรียมไว้ และทำให้ระบบทำงานได้บนเครื่องของผู้พัฒนา

## 1. เป้าหมายของบทนี้

เมื่อทำตามบทนี้จบ ควรทำได้ดังนี้

- clone โปรเจกต์จาก GitHub ลงเครื่องได้
- ติดตั้ง Node.js และ dependencies ได้ถูกต้อง
- สร้างไฟล์ environment สำหรับ Supabase ได้
- เข้าใจว่าไฟล์ใดเป็นจุดเชื่อม Supabase, Auth, Search และ AI
- รันระบบด้วย `npm run dev` ได้
- เปิดหน้า public และหน้า login ได้
- login ด้วยผู้ใช้ที่สร้างใน Supabase ได้
- ทดสอบ build ก่อน deploy ได้
- แก้ปัญหาเบื้องต้น เช่น env ไม่ขึ้น, login ไม่ได้, build fail, port ชน ได้

ระบบนี้เป็น frontend แบบ Single Page Application พัฒนาด้วย React และ Vite

| Build tool | Vite | รัน dev server และ build ไฟล์ production |

| Route | React Router | จัดการเส้นทางหน้า public และ dashboard |
| Data cache | TanStack React Query | cache ข้อมูลและลดการยิง request ซ้ำ |
| Backend | Supabase | Database, Auth, RLS และ RPC |

## 5. Clone โปรเจกต์จาก GitHub

เลือกโฟลเดอร์ที่ต้องการเก็บโปรเจกต์ เช่น `D:\projects` หรือ `C:\Users\<ชื่อผู้ใช้>\Documents\projects`

ตัวอย่างบน Windows PowerShell

```powershell
cd D:\projects
git clone https://github.com/dragonfly13110/npt_dashboard.git
cd npt_dashboard
```

ตัวอย่างบน macOS หรือ Linux

```bash
cd ~/projects
git clone https://github.com/dragonfly13110/npt_dashboard.git
cd npt_dashboard
```

ตรวจว่าอยู่ใน repo ถูกต้อง

```bash
git status
```

ถ้าขึ้นประมาณนี้ถือว่าเข้า repo ถูกแล้ว

```text
On branch main
nothing to commit, working tree clean
```

ถ้า clone ไม่ได้ ให้ตรวจ 3 จุดนี้

- login GitHub ถูกบัญชีหรือไม่
- บัญชีนั้นมีสิทธิ์อ่าน repo หรือไม่
- ใช้ HTTPS แล้วติด credential เก่าหรือไม่

## 6. ติดตั้ง dependencies

โปรเจกต์นี้ใช้ npm script ใน `package.json` ดังนั้นเริ่มจาก `npm install`

```bash
npm install
```

หลังติดตั้งเสร็จ จะมีโฟลเดอร์ `node_modules/` เพิ่มขึ้นมาในเครื่อง

ไม่ต้อง commit `node_modules/` เพราะถูก ignore ไว้แล้ว

ถ้าติดตั้งแล้วมี warning บางรายการ ให้ดูระดับความรุนแรงก่อน

| อาการ | ความหมาย | ทำอย่างไร |
|---|---|---|
| `npm WARN deprecated` | package บางตัวมีคำเตือน | ยังรันได้ ให้บันทึกไว้ก่อน |
| `npm ERR!` | ติดตั้งไม่สำเร็จ | อ่าน error แล้วแก้ตามสาเหตุ |
| ติด permission | npm เขียนไฟล์ไม่ได้ | ปิด terminal เปิดใหม่ หรือย้ายโปรเจกต์ไปโฟลเดอร์ที่มีสิทธิ์ |
| node version ต่ำ | package ใช้ Node รุ่นใหม่ | อัปเดต Node.js เป็น LTS ใหม่ |

คำสั่งล้างติดตั้งใหม่ ถ้าติดปัญหาแปลก ๆ

```bash
rm -rf node_modules package-lock.json
npm install
```

บน Windows PowerShell

```powershell
Remove-Item -Recurse -Force node_modules
Remove-Item -Force package-lock.json
npm install
```

หมายเหตุ: ใช้คำสั่งลบ `package-lock.json` เฉพาะกรณีติดตั้งเสียหรือ dependency เพี้ยนจริง ๆ ถ้าทีมเริ่มใช้ lock file ร่วมกันแล้ว ไม่ควรลบเองโดยไม่แจ้งทีม

## 7. ตั้งค่าไฟล์ environment

โปรเจกต์ใช้ environment variables เพื่อบอก frontend ว่าต้องเชื่อม Supabase project ใด

ไฟล์ตัวอย่างคือ

```text
.env.example
```

ในไฟล์ตัวอย่างมีค่าหลัก

```env
VITE_SUPABASE_URL=https://your-project.supabase.co
VITE_SUPABASE_ANON_KEY=your-anon-key-here
```

### 7.1 สร้างไฟล์ `.env.local`

แนะนำให้ใช้ `.env.local` สำหรับเครื่องของตัวเอง

Windows PowerShell

```powershell
Copy-Item .env.example .env.local
```

macOS หรือ Linux

```bash
cp .env.example .env.local
```

จากนั้นเปิด `.env.local` แล้วแก้ค่าให้ตรงกับ Supabase project จริง

```env
VITE_SUPABASE_URL=https://xxxxxxxxxxxxxxxxxxxx.supabase.co
VITE_SUPABASE_ANON_KEY=ใส่ค่า-anon-key-ของ-project
```

### 7.2 หา Supabase URL และ anon key จากที่ไหน

ใน Supabase Dashboard

```text
Project Settings
  → API
    → Project URL
    → Project API keys
      → anon public
```

นำค่า `Project URL` มาใส่ใน `VITE_SUPABASE_URL`

นำค่า `anon public` มาใส่ใน `VITE_SUPABASE_ANON_KEY`

### 7.3 ข้อควรระวังเรื่อง key

- ใช้ `anon key` ใน frontend ได้ เพราะถูกออกแบบให้ใช้ร่วมกับ RLS
- ห้ามใส่ `service_role key` ในไฟล์ frontend หรือไฟล์ที่อาจถูก commit
- ห้าม commit `.env`, `.env.local`, `.env.production` ขึ้น GitHub
- ไฟล์ `.gitignore` ของโปรเจกต์ ignore `.env` และ `.env.*` ไว้แล้ว แต่ควรตรวจ `git status` ทุกครั้งก่อน commit
- ถ้าค่า env เปลี่ยน ต้องหยุด dev server แล้วรันใหม่

### 7.4 หมายเหตุเรื่อง fallback ใน `src/supabaseClient.js`

ในโค้ดปัจจุบัน `src/supabaseClient.js` มีค่า fallback สำหรับ Supabase URL และ anon key เพื่อให้ระบบยังพอรันได้ถ้าไม่ได้ตั้ง env

ในการใช้งานจริงควรตั้ง `.env.local` ทุกครั้ง เพราะจะทำให้รู้แน่นอนว่าเครื่องนี้กำลังเชื่อมกับ Supabase project ใด

ถ้าเห็นข้อความเตือนใน console ว่าไม่พบ `VITE_SUPABASE_URL` ให้ถือว่า env ยังไม่ถูกตั้งหรือ dev server ยังไม่ได้ restart

## 8. การตั้งค่า Supabase ให้พร้อมก่อนรัน frontend

ก่อนรันเว็บ ควรทำ Supabase ให้พร้อมตามบท 04 ก่อน

ลำดับที่แนะนำ

```text
1. สร้าง Supabase project
2. รัน SQL หลักใน supabase/schema.sql
3. รัน migration ที่เกี่ยวข้อง เช่น migration_rbac_audit.sql
4. รัน data_requests.sql ถ้าจะใช้ระบบคำขอข้อมูล
5. รัน SQL ตารางเสริมที่ต้องใช้ เช่น forecast_plots.sql
6. สร้าง user สำหรับทดสอบใน Supabase Auth
7. ตั้ง role และ department ในตาราง profiles
8. ตรวจ RLS policy ก่อนนำข้อมูลจริงเข้า
```

### 8.1 ตรวจตารางสำคัญ

หลังรัน SQL แล้ว ควรมีตารางสำคัญ เช่น

| กลุ่ม | ตารางตัวอย่าง |
|---|---|
| ระบบผู้ใช้ | `profiles`, `audit_logs` |
| บริหาร | `personnel`, `assets`, `budgets` |
| ยุทธศาสตร์ | `farmer_registry`, `gis_areas`, `agricultural_areas`, `learning_centers`, `daily_weather` |
| ส่งเสริมการผลิต | `large_plots`, `certifications`, `crop_production` |
| พัฒนาเกษตรกร | `community_enterprises`, `smart_farmers`, `farmer_groups`, `farmer_institutes`, `agri_tourism` |
| อารักขาพืช | `forecast_plots`, `pest_centers`, `soil_fertilizer_centers`, `fire_hotspots`, `pest_outbreaks` |
| คำขอข้อมูล | `data_requests`, `data_request_assignments`, `data_request_responses` |

### 8.2 ตั้ง admin คนแรก

ถ้าสร้าง user แล้ว login ได้ แต่เข้าเมนูจัดการผู้ใช้ไม่ได้ ให้ตรวจ `profiles.role`

ตัวอย่าง SQL

```sql
UPDATE profiles
SET
  role = 'admin',
  full_name = 'ผู้ดูแลระบบ',
  department = 'ฝ่ายบริหารทั่วไป',
  updated_at = NOW()
WHERE email = 'your-email@example.com';
```

ตรวจผล

```sql
SELECT id, email, full_name, department, role
FROM profiles
ORDER BY created_at DESC;
```

### 8.3 ตั้ง editor สำหรับกลุ่มงาน

ตัวอย่าง

```sql
UPDATE profiles
SET
  role = 'editor',
  full_name = 'เจ้าหน้าที่กลุ่มส่งเสริมและพัฒนาการผลิต',
  department = 'กลุ่มส่งเสริมและพัฒนาการผลิต',
  updated_at = NOW()
WHERE email = 'production@example.com';
```

### 8.4 ตั้ง editor สำหรับอำเภอในระบบ Data Request

ถ้าจะให้สำนักงานเกษตรอำเภอตอบคำขอข้อมูล ให้ตั้ง `department` เป็นชื่ออำเภอให้ตรงกับข้อมูล assignment

```sql
UPDATE profiles
SET
  role = 'editor',
  full_name = 'สำนักงานเกษตรอำเภอพุทธมณฑล',
  department = 'พุทธมณฑล',
  updated_at = NOW()
WHERE email = 'phutthamonthon@example.com';
```

## 10. หน้าที่ควรเปิดตรวจหลังรันระบบ

หลัง dev server เปิดแล้ว ควรตรวจทีละหน้า

| URL | ควรเห็นอะไร | ใช้ตรวจอะไร |
|---|---|---|
| `/` | หน้า landing page | public route และ asset พื้นฐาน |
| `/interactive-dashboard` | dashboard สาธารณะ | chart, widget, public data |
| `/public/large-plots` | ข้อมูลแปลงใหญ่ public | public data page |
| `/login` | หน้าเข้าสู่ระบบ | Supabase Auth flow |
| `/dashboard` | dashboard ภายใน | protected route หลัง login |
| `/dashboard/search` | หน้าค้นหา | global search service |
| `/dashboard/chatbot` | chatbot | AI proxy และข้อมูลประกอบคำตอบ |
| `/dashboard/admin/users` | จัดการผู้ใช้ | admin route |
| `/dashboard/data-requests` | คำขอข้อมูล | role admin/editor |

ถ้าเปิด `/dashboard` แล้วถูกส่งกลับ `/login` แปลว่าระบบมองว่ายังไม่ได้ login

ถ้า login แล้วเข้า `/dashboard/admin/users` ไม่ได้ ให้ตรวจว่า role เป็น `admin` หรือไม่

## 11. ทำความเข้าใจ route และสิทธิ์หน้าเว็บ

route หลักถูกกำหนดใน `src/App.jsx`

ระบบแบ่ง route เป็น 3 กลุ่มใหญ่

| กลุ่ม route | ตัวอย่าง | เงื่อนไข |
|---|---|---|
| Public | `/`, `/interactive-dashboard`, `/public/*` | ไม่ต้อง login |
| Protected | `/dashboard/*` | ต้อง login หรือมี session |
| Admin only | `/dashboard/admin/users`, `/dashboard/admin/audit-log` | ต้องเป็น admin |

ใน `App.jsx` มี helper route สำคัญ

| helper | ใช้ทำอะไร |
|---|---|
| `ProtectedRoute` | ถ้ายังไม่ login ให้ส่งไป `/login` |
| `AdminRoute` | ถ้าไม่ใช่ admin ให้ส่งกลับ `/dashboard` |
| `DataRequestRoute` | อนุญาตเฉพาะ `admin` และ `editor` |
| `NonGuestRoute` | กัน guest ไม่ให้เข้า chatbot |

สิทธิ์หน้าเว็บช่วยเรื่องประสบการณ์ใช้งาน แต่ความปลอดภัยจริงต้องอยู่ที่ Supabase RLS ด้วย

## 12. ทำความเข้าใจ Auth และ profile

ระบบ login ใช้ Supabase Auth แต่ข้อมูลสิทธิ์ของผู้ใช้จะอ่านจากตาราง `profiles`

flow โดยย่อ

```text
ผู้ใช้ login
  ↓
Supabase Auth คืน session
  ↓
AuthContext อ่าน user.id
  ↓
AuthContext ดึง row จาก profiles
  ↓

  ↓
```

ไฟล์สำคัญคือ

| role | ความหมายในหน้าเว็บ |
|---|---|
| `guest` | โหมดผู้ใช้ทั่วไปหรือ public |
| `viewer` | ดูข้อมูลภายในตามสิทธิ์ |
| `editor` | เพิ่ม/แก้ไขข้อมูลบางส่วนได้ |
| `admin` | จัดการระบบและผู้ใช้ได้ |

ตัวอย่าง department ที่ frontend map เป็น group

| department | group key |
|---|---|
| `ฝ่ายบริหารทั่วไป` | `admin` |
| `กลุ่มยุทธศาสตร์และสารสนเทศ` | `strategy` |
| `กลุ่มส่งเสริมและพัฒนาการผลิต` | `production` |
| `กลุ่มส่งเสริมและพัฒนาเกษตรกร` | `development` |
| `กลุ่มอารักขาพืช` | `protection` |

## 14. การตั้งค่า proxy ใน Vite

ไฟล์ `vite.config.js` ตั้งค่า proxy เพื่อให้ frontend เรียก API ภายนอกหรือ function ได้ง่ายขึ้นระหว่างพัฒนา

proxy สำคัญที่มีในระบบ เช่น

| path ใน frontend | ปลายทาง | ใช้ทำอะไร |
|---|---|---|
| `/.netlify/functions` | deployed Netlify site | เรียก Netlify Functions เช่น AI proxy |
| `/api/nabc` | `agriapi.nabc.go.th` | ข้อมูลจาก NABC |
| `/api/gistda` | `api-gateway.gistda.or.th` | ข้อมูล GISTDA |
| `/api/doae-npt` | `nakhonpathom.doae.go.th` | ข่าวหรือข้อมูลจากเว็บสำนักงานจังหวัด |
| `/api/doae-esc` | `esc.doae.go.th` | ข้อมูลจาก ESC |
| `/api/doae-hq` | `www.doae.go.th` | ข้อมูลจาก DOAE ส่วนกลาง |
| `/api/rss/*` | แหล่ง RSS หลายเว็บ | ข่าวเกษตร |

ข้อควรระวัง

- ถ้า API ภายนอกล่ม หน้า widget ที่ใช้ API นั้นอาจไม่แสดงข้อมูล แต่ส่วน Supabase ยังทำงานได้
- ถ้าใช้ GISTDA จริง ควรตั้ง `VITE_GISTDA_API_KEY` ใน env แทนการพึ่ง fallback
- ถ้าจะทดสอบ Netlify Functions ในเครื่องแบบเต็ม ควรใช้ Netlify CLI และ `netlify dev` เพิ่มเติมในขั้นตอน deploy หรือบท 07

## 15. การตั้งค่า AI Chatbot สำหรับ development

AI Chatbot เรียกผ่าน function

```text
/.netlify/functions/ai-proxy
```

ไฟล์ที่เกี่ยวข้อง

| ไฟล์ | หน้าที่ |
|---|---|
| `src/pages/Chatbot.jsx` | หน้า chatbot |
| `src/services/aiService.js` | ส่งคำถามไปยัง AI proxy |
| `src/services/chatbotDataService.js` | เตรียมข้อมูลจาก Supabase ให้ AI |

AI proxy รองรับ provider หลัก เช่น

| provider | env ที่เกี่ยวข้อง | หมายเหตุ |
|---|---|---|

ข้อควรระวังเรื่อง key

- API key ของ AI ไม่ควรอยู่ใน frontend
- ควรตั้ง key ใน Netlify environment variables
- ระหว่าง dev ถ้าใช้ proxy ไปยัง deployed function ต้องตรวจว่า deployed site มี env แล้ว
- ถ้า chatbot แจ้ง `API key not configured` ให้ตรวจ env ใน Netlify หรือ local function environment

## 22. การตรวจระบบ Search

Global Search อยู่ที่

```text
/dashboard/search
```

ไฟล์สำคัญคือ

```text
src/services/globalSearchService.js
src/utils/chatbotConstants.js
```

ลำดับการทำงานของ Search

```text
ผู้ใช้ค้นหา
  ↓
เรียก globalSearch(query)
  ↓
ลองเรียก Supabase RPC: global_search
  ↓
ถ้า RPC สำเร็จ ใช้ผลลัพธ์จาก RPC
  ↓
ถ้า RPC fail ใช้ fallback แบบ parallel query หลายตาราง
  ↓
แสดงผลตาม TABLE_CONFIG และ TABLE_ROUTES
```

ถ้า search ไม่ทำงาน ให้ตรวจ

- มี function `global_search` ใน Supabase หรือไม่
- table ที่ต้องการค้นหาถูกเพิ่มใน `TABLE_CONFIG` หรือไม่
- table นั้นมี `TABLE_SEARCH_COLS` หรือไม่
- RLS อนุญาตให้ user ค้นหาตารางนั้นหรือไม่
- column ที่ใช้ค้นหามีอยู่จริงหรือไม่

ถ้าเพิ่มตารางใหม่ ต้องเพิ่มอย่างน้อย 3 จุด

```text
1. Supabase table
2. src/utils/chatbotConstants.js
3. src/services/globalSearchService.js หรือ config route ที่เกี่ยวข้อง
```

## 28. ปัญหาที่พบบ่อยและวิธีแก้

### 28.1 `npm install` ไม่ผ่าน

อาการ

```text
npm ERR!
```

แนวทางแก้

1. ตรวจ Node.js version
2. ลบ `node_modules`
3. ลบ cache ถ้าจำเป็น
4. รัน `npm install` ใหม่

```bash
npm cache verify
npm install
```

ถ้ายังไม่ผ่าน ให้ส่ง error เต็ม ๆ ให้ทีมดู ไม่ควรเดาแล้วลบไฟล์สำคัญหลายชุดพร้อมกัน

### 28.2 `npm run dev` เปิดไม่ได้

ตรวจตามนี้

- port ถูกใช้อยู่หรือไม่
- terminal อยู่ในโฟลเดอร์ repo หรือไม่
- ติดตั้ง dependencies แล้วหรือยัง
- มี syntax error จากไฟล์ที่เพิ่งแก้หรือไม่

เปลี่ยน port ได้ด้วย

```bash
npm run dev -- --port 5174
```

### 28.3 หน้าเว็บขาวหรือ error ใน browser

ให้เปิด Developer Tools แล้วดู Console

จุดที่ควรดู

- import path ผิด
- component error
- env ไม่พบ
- Supabase request fail
- CORS หรือ network error

ถ้ามี error จาก component ให้ดู stack trace ว่าชี้ไปไฟล์ใด

### 28.4 login ไม่ได้

ตรวจตามลำดับ

- email/password ถูกต้องหรือไม่
- user ถูกสร้างใน Supabase Auth หรือยัง
- email confirmation เปิดอยู่หรือไม่
- Supabase URL และ anon key ถูก project หรือไม่
- RLS policy ของ `profiles` อนุญาตให้อ่านหรือไม่

### 28.5 login ได้แต่ไม่มีเมนูหรือเข้า dashboard ไม่ได้

ตรวจ `profiles`

```sql
SELECT id, email, full_name, department, role
FROM profiles
WHERE email = 'your-email@example.com';
```

ตรวจว่า

- role เป็น `admin`, `editor` หรือ `viewer`
- department สะกดตรงกับ frontend
- row ใน profiles ผูกกับ user id ถูกต้อง

### 28.6 ข้อมูลไม่ขึ้นในตาราง

ตรวจตามนี้

- ตารางใน Supabase มีข้อมูลจริงหรือไม่
- column ชื่อตรงกับ frontend หรือไม่
- RLS block หรือไม่
- user role/department มีสิทธิ์อ่านหรือไม่
- มี error ใน console หรือ network หรือไม่

### 28.7 Search ไม่เจอข้อมูล

ตรวจ

- `global_search` RPC มีอยู่หรือไม่
- ถ้า RPC ไม่มี fallback ทำงานหรือไม่
- table ถูกใส่ใน `TABLE_CONFIG` หรือไม่
- column ถูกใส่ใน `TABLE_SEARCH_COLS` หรือไม่
- คำค้นสั้นเกินไปหรือไม่ ระบบต้องการอย่างน้อย 2 ตัวอักษร

### 28.8 Chatbot ใช้ไม่ได้

ตรวจ

- login เป็น guest อยู่หรือไม่
- Netlify Function ตอบกลับหรือไม่
- API key ใน Netlify ตั้งแล้วหรือยัง
- provider ที่เลือกตรงกับ function หรือไม่
- Supabase query ที่ chatbot ใช้โดน RLS block หรือไม่

### 28.9 build fail

รัน

```bash
npm run build
```

แล้วอ่าน error บรรทัดแรกที่เป็นสาเหตุจริง

สาเหตุที่พบบ่อย

- import ไฟล์ผิดตัวพิมพ์เล็กใหญ่
- ใช้ตัวแปรที่ไม่ได้ประกาศ
- syntax JSX ผิด
- package ยังไม่ได้ install
- environment ที่ build ไม่มีค่าจำเป็น

### 28.10 refresh หน้า `/dashboard/...` แล้ว 404 บน production

ตรวจ `netlify.toml` ว่ามี redirect นี้หรือไม่

```toml
[[redirects]]
  from = "/*"
  to = "/index.html"
  status = 200
```

ถ้ามีแล้วแต่ยัง 404 ให้ตรวจว่า Netlify ใช้ไฟล์ `netlify.toml` จาก repo ล่าสุดหรือไม่

## 29. แนวทางเพิ่มตารางหรือโมดูลใหม่ในอนาคต

ถ้าจะเพิ่มตารางใหม่ เช่น `organic_farms` ไม่ควรเพิ่มเฉพาะหน้าเว็บ ต้องเพิ่มให้ครบทั้งระบบ

ลำดับที่แนะนำ

```text
1. ออกแบบ table ใน Supabase
2. สร้าง SQL migration
3. เปิด RLS และ policy
4. เพิ่มข้อมูลตัวอย่าง
5. สร้างหน้า React หรือเพิ่มเข้า Data Management
6. เพิ่ม route ใน App.jsx
7. เพิ่มเมนูใน Layout ถ้ามี
8. เพิ่ม TABLE_CONFIG ใน chatbotConstants.js
9. เพิ่ม TABLE_SEARCH_COLS ถ้าต้องการค้นหา
10. เพิ่ม route mapping ใน globalSearchService.js
11. เพิ่ม logic ใน chatbotDataService.js ถ้าต้องให้ AI อ่านข้อมูล
12. ทดสอบ login, search, chatbot, build
```

ห้ามเพิ่ม field หรือ table โดยไม่มีเจ้าของข้อมูลและระดับการเผยแพร่ เพราะจะทำให้ RLS และหน้า public เสี่ยงผิดพลาดภายหลัง
