เอกสารนี้เป็นเนื้อหาต้นฉบับสำหรับนำไปออกแบบ infographic แบบ 4 ตอน ครอบคลุมที่มา หลักการทำงาน โครงสร้าง วิธีใช้งาน ผลลัพธ์ และประโยชน์ของระบบ `NPT Smart Agri Dashboard`

## ตอนที่ 1: ที่มาและโจทย์ของระบบ

### ชื่อหลัก

จากข้อมูลเกษตรที่กระจัดกระจาย สู่ศูนย์ข้อมูลกลางระดับจังหวัด

### ประเด็นเปิดเรื่อง

งานด้านการเกษตรระดับจังหวัดมีข้อมูลจำนวนมากจากหลายกลุ่มงาน หลายพื้นที่ และหลายแหล่งข้อมูล เช่น ข้อมูลเกษตรกร พื้นที่เพาะปลูก แปลงใหญ่ วิสาหกิจชุมชน Smart Farmer ภัยพิบัติ จุดความร้อน สภาพอากาศ ราคาสินค้าเกษตร และข้อมูลจากหน่วยงานภายนอก

เดิมข้อมูลเหล่านี้มักอยู่ในหลายไฟล์ หลายตาราง หลายระบบ หรืออยู่กับเจ้าหน้าที่แต่ละกลุ่มงาน ทำให้การค้นหา การรวมข้อมูล และการทำรายงานต้องใช้เวลามาก

### ปัญหาเดิม

- ข้อมูลกระจายอยู่หลายแหล่ง ไม่ได้รวมในจุดเดียว
- เจ้าหน้าที่ต้องเปิดหลายไฟล์หรือหลายระบบเพื่อหาคำตอบหนึ่งเรื่อง
- การทำรายงานภาพรวมจังหวัดใช้เวลานาน
- ผู้บริหารเห็นสถานการณ์ช้ากว่าที่ควร
- ข้อมูลบางส่วนเผยแพร่ต่อประชาชนได้ยาก
- ข้อมูลภาคสนามหรือข้อมูลจากอำเภอมีรูปแบบไม่เหมือนกัน
- การตรวจสอบข้อมูลย้อนหลังทำได้ลำบาก
- ข้อมูลเชิงพื้นที่ เช่น อำเภอ พิกัด หรือ hotspot ยังไม่ถูกนำมาใช้ร่วมกับ dashboard อย่างเต็มที่

### โจทย์ที่ต้องแก้

ระบบต้องช่วยให้จังหวัดตอบคำถามสำคัญได้เร็วขึ้น เช่น

- พื้นที่เกษตรแต่ละอำเภอมีเท่าไร
- แปลงใหญ่อยู่ที่ไหนบ้าง
- Smart Farmer และ Young Smart Farmer กระจายอยู่พื้นที่ใด
- วิสาหกิจชุมชน กลุ่มเกษตรกร และแหล่งท่องเที่ยวเชิงเกษตรมีข้อมูลอย่างไร
- มีภัยพิบัติ จุดความร้อน หรือพื้นที่เฝ้าระวังตรงไหน
- ข้อมูลใดควรเผยแพร่ให้ประชาชนเห็น
- ข้อมูลใดควรจำกัดเฉพาะเจ้าหน้าที่
- จะลดเวลาการรวบรวมข้อมูลจาก Excel และเอกสารแยกได้อย่างไร

### แนวคิดของระบบ

`NPT Smart Agri Dashboard` ถูกออกแบบให้เป็นศูนย์กลางข้อมูลเกษตรของจังหวัดนครปฐม โดยรวมข้อมูลจากหลายกลุ่มงานไว้ในระบบเดียว และแสดงผลในรูปแบบที่ใช้งานง่าย ทั้งตาราง dashboard กราฟ แผนที่ การค้นหา และผู้ช่วย AI

ระบบไม่ได้เป็นเพียงหน้าเว็บแสดงข้อมูล แต่เป็นแพลตฟอร์มสำหรับจัดการข้อมูล วิเคราะห์ข้อมูล และสนับสนุนการตัดสินใจ

### กลุ่มผู้ใช้งาน

- ประชาชน: ดูข้อมูลเกษตรที่เผยแพร่สาธารณะ
- เจ้าหน้าที่: ดู ค้นหา เพิ่ม แก้ไข และจัดการข้อมูลตามกลุ่มงาน
- ผู้บริหาร: ดูภาพรวม แนวโน้ม และข้อมูลประกอบการตัดสินใจ
- ผู้ดูแลระบบ: จัดการผู้ใช้ สิทธิ์ ข้อมูลหลัก และ audit log
- อำเภอหรือหน่วยงานภาคสนาม: ส่งข้อมูลกลับเข้าสู่ระบบผ่านกระบวนการ Data Requests

### เป้าหมายหลัก

- รวมข้อมูลเกษตรจังหวัดไว้ในจุดเดียว
- ลดเวลารวบรวมข้อมูลและทำรายงาน
- ทำให้ข้อมูลค้นหาและตรวจสอบได้ง่าย
- แสดงภาพรวมผ่าน dashboard และแผนที่
- ใช้ AI ช่วยสรุปและตอบคำถามจากข้อมูลจริง
- แยกข้อมูล public และ internal อย่างชัดเจน
- รองรับการต่อยอดเป็นระบบข้อมูลเกษตรจังหวัดที่ขยายผลได้

### ข้อความสั้นสำหรับใช้บน infographic

ข้อมูลเดิม: กระจัดกระจาย ค้นหายาก ทำรายงานช้า  
ระบบใหม่: รวมศูนย์ ค้นหาเร็ว เห็นภาพรวม ใช้ AI ช่วยสรุป

## ตอนที่ 2: หลักการทำงานและกระบวนการข้อมูล

### ชื่อหลัก

ข้อมูลไหลจากหลายแหล่ง เข้าระบบกลาง แล้วแสดงผลเป็น dashboard, map, search และ AI

### ภาพรวมการทำงาน

ระบบทำงานแบบเว็บแอปพลิเคชัน ผู้ใช้เข้าใช้งานผ่านหน้าเว็บ จากนั้นระบบจะจัดการเส้นทางการใช้งาน สิทธิ์ผู้ใช้ การดึงข้อมูล การ cache ข้อมูล และการแสดงผลผ่านโมดูลต่าง ๆ

โครงสร้างการไหลของข้อมูล:

```text
ผู้ใช้
  -> React Web App
  -> Auth / Route / Dashboard / Search / Chatbot
  -> Service Layer
  -> Supabase / AI Proxy / External APIs
  -> ตาราง / กราฟ / แผนที่ / คำตอบ AI
```

### แหล่งข้อมูลหลัก

ระบบรองรับข้อมูลหลายประเภท ได้แก่

- ฐานข้อมูล Supabase สำหรับข้อมูลหลักของระบบ
- Supabase Auth สำหรับ login และจัดการ session
- ตารางข้อมูลเกษตรตามกลุ่มงาน
- RPC สำหรับการค้นหาข้ามตาราง
- Netlify Functions สำหรับ proxy ไปยัง API ภายนอก
- External APIs เช่น ข่าว RSS สภาพอากาศ AQI ราคาเกษตร hotspot และข้อมูลหน่วยงานอื่น

### ขั้นตอนการทำงานของ Dashboard

1. ผู้ใช้เปิดหน้า dashboard หรือหน้า public
2. ระบบตรวจสอบว่าหน้านั้นเป็น public หรือ protected
3. ถ้าเป็น protected จะตรวจ session และ role
4. React Query หรือ custom hook เรียกข้อมูลจาก Supabase
5. ระบบรวมข้อมูลจากหลายตาราง
6. แปลงข้อมูลเป็นรูปแบบที่หน้า UI ใช้ได้
7. แสดงผลเป็นตัวเลขสรุป ตาราง กราฟ และแผนที่
8. Cache ข้อมูลไว้เพื่อลดการเรียกซ้ำ

### ขั้นตอนการทำงานของระบบค้นหา

1. ผู้ใช้พิมพ์คำค้น
2. ระบบส่งคำค้นไปยัง `globalSearchService`
3. ระบบพยายามใช้ Supabase RPC `global_search`
4. ถ้า RPC ใช้ไม่ได้ จะ fallback เป็นการ query หลายตารางแบบขนาน
5. รวมผลลัพธ์จากหลายหมวดข้อมูล
6. แสดงผลในหน้า Search Results
7. เก็บ recent searches และ cache สั้น ๆ เพื่อให้ค้นซ้ำเร็วขึ้น

### ขั้นตอนการทำงานของ AI Chatbot

AI Chatbot มีแนวคิดสำคัญคือ ใช้ข้อมูลจริงจากฐานข้อมูลก่อน แล้วให้ AI ช่วยสรุป ไม่ปล่อยให้ AI ตอบจากความจำของโมเดลอย่างเดียว

Flow การทำงาน:

```text
คำถามผู้ใช้
  -> วิเคราะห์ intent
  -> ระบุหัวข้อ / ตาราง / อำเภอ / keyword ที่เกี่ยวข้อง
  -> ดึงข้อมูลจาก Supabase
  -> คำนวณตัวเลขสรุป เช่น count, total, average, ranking, by district
  -> ส่ง context ที่ย่อแล้วให้ AI
  -> AI สร้างคำตอบ
  -> แสดงคำตอบพร้อมบริบทข้อมูล
```

### ตัวอย่างคำถามที่ระบบตอบได้

- อำเภอไหนมีแปลงใหญ่มากที่สุด
- สรุปข้อมูล Smart Farmer ในจังหวัด
- พื้นที่เกษตรแต่ละอำเภอเป็นอย่างไร
- วิสาหกิจชุมชนมีข้อมูลอะไรบ้าง
- มีจุด hotspot หรือพื้นที่เฝ้าระวังตรงไหน
- ข้อมูล Young Smart Farmer แยกตามพื้นที่เป็นอย่างไร
- ข้อมูลการสำรวจมะพร้าวน้ำหอมมีภาพรวมอย่างไร

### หลักการจัดการสิทธิ์

ระบบแยกการเข้าถึงตามประเภทผู้ใช้

- Public route: เปิดให้ประชาชนเข้าถึงได้
- Protected route: ต้อง login ก่อนเข้าใช้งาน
- Admin route: เฉพาะผู้ดูแลระบบ
- Data Request route: เฉพาะ admin และ editor
- Non-guest route: จำกัดบางฟังก์ชันไม่ให้ guest ใช้

### หลักการ cache และประสิทธิภาพ

ระบบใช้ cache หลายระดับ

- React Query cache สำหรับ dashboard และข้อมูลหลัก
- stale time ประมาณ 15 นาที
- garbage collection ประมาณ 60 นาที
- in-memory cache สำหรับ global search
- lazy loading หน้าแต่ละส่วนเพื่อลดขนาดโหลดแรก
- Netlify Functions ช่วยเป็น proxy เพื่อลดปัญหาเรียก API จาก frontend โดยตรง

### ข้อความสั้นสำหรับใช้บน infographic

ระบบไม่ใช่แค่เก็บข้อมูล แต่เปลี่ยนข้อมูลให้เป็นคำตอบ  
Dashboard บอกภาพรวม Search หาข้อมูลเร็ว AI ช่วยสรุป Map เห็นพื้นที่จริง

## ตอนที่ 3: โครงสร้างระบบและโมดูลสำคัญ

### ชื่อหลัก

สถาปัตยกรรมแบบแยกชั้น: ใช้ง่าย ดูแลง่าย ขยายต่อได้

### Technology Stack

- Frontend: React 19 + Vite
- Routing: React Router
- UI Framework: Ant Design
- State และ Data Cache: TanStack React Query
- Database และ Auth: Supabase
- Chart: Recharts
- Map: Leaflet และ React-Leaflet
- AI Integration: AI Proxy ผ่าน Netlify Functions รองรับ Gemini / OpenRouter
- File Utilities: xlsx, jsPDF, html2canvas
- Testing: Vitest, Playwright, Testing Library
- Deploy: Netlify

### Layer ของระบบ

#### 1. Presentation Layer

หน้าที่คือแสดงผลและรับ interaction จากผู้ใช้

องค์ประกอบหลัก:

- หน้า Landing Page
- Interactive Dashboard
- หน้า public detail
- Internal Dashboard
- CRUD pages
- Search Results
- Chatbot
- Layout, Sidebar, Widget, Table, Map, Chart

#### 2. Provider และ State Layer

หน้าที่คือจัดการสถานะกลางของแอป

องค์ประกอบหลัก:

- `AuthProvider`: จัดการ user, profile, role, department
- `QueryClientProvider`: จัดการ cache และสถานะ loading/error
- `ConfigProvider`: ตั้งค่า theme และภาษาไทยของ Ant Design
- Error Boundary: ป้องกันหน้าเว็บล่มทั้งระบบเมื่อ component บางส่วน error

#### 3. Service Layer

หน้าที่คือเชื่อมต่อข้อมูลและ logic กลาง

องค์ประกอบหลัก:

- `globalSearchService`: ค้นหาข้ามหลายตาราง
- `chatbotDataService`: เตรียมข้อมูลจากฐานข้อมูลสำหรับ AI
- `aiService`: เรียก AI provider ผ่าน proxy
- Supabase client: ติดต่อฐานข้อมูลและ auth
- utility ต่าง ๆ เช่น audit log, geo helper, export helper

#### 4. Data Layer

หน้าที่คือเก็บข้อมูลจริงและจัดการสิทธิ์ระดับฐานข้อมูล

องค์ประกอบหลัก:

- Supabase database
- Supabase Auth
- ตารางข้อมูลเกษตร
- SQL migration และ schema ในโฟลเดอร์ `supabase`
- RPC เช่น global search
- RLS / policies สำหรับความปลอดภัยฝั่งฐานข้อมูล

#### 5. Serverless / Proxy Layer

หน้าที่คือเป็นตัวกลางเรียก API ภายนอกหรือ AI

ตัวอย่าง Netlify Functions:

- `ai-proxy.js`
- `rss-proxy.js`
- `gistda-proxy.js`
- `moc-price-proxy.js`
- `sync-weather.js`
- `sync-hotspots.js`
- `doae-npt-proxy.js`
- `wp-proxy.js`

## ตอนที่ 4: วิธีใช้งาน ผลลัพธ์ และประโยชน์

### ชื่อหลัก

จากฐานข้อมูลกลาง สู่การตัดสินใจที่เร็วขึ้นและโปร่งใสขึ้น

### วิธีใช้งานตามบทบาท

#### ประชาชน

1. เข้าเว็บไซต์
2. ดูหน้า Landing Page
3. เปิด Interactive Dashboard
4. ดูข้อมูล public เช่น แปลงใหญ่ Smart Farmer วิสาหกิจชุมชน ท่องเที่ยวเกษตร พื้นที่เกษตร ข่าว ราคา อากาศ AQI และ hotspot
5. ใช้ข้อมูลเพื่อรับรู้สถานการณ์เกษตรจังหวัด

#### เจ้าหน้าที่

1. Login เข้าระบบ
2. เข้า dashboard ตามกลุ่มงาน
3. ดูข้อมูลในตาราง
4. เพิ่ม แก้ไข ลบ หรือนำเข้าข้อมูล
5. ใช้ global search ค้นหาข้อมูลข้ามหมวด
6. ใช้ chatbot ถามสรุปข้อมูล
7. ส่งออกข้อมูลหรือเตรียมรายงาน

#### ผู้บริหาร

1. ดู dashboard ภาพรวม
2. ดูตัวชี้วัด แผนที่ กราฟ และสถานการณ์ล่าสุด
3. ตรวจข้อมูลตามอำเภอหรือกลุ่มงาน
4. ใช้ AI ช่วยสรุปประเด็นสำคัญ
5. ใช้ข้อมูลประกอบการประชุมหรือการตัดสินใจ

#### ผู้ดูแลระบบ

1. จัดการผู้ใช้
2. กำหนด role และ department
3. ตรวจ audit log
4. ตรวจข้อมูลการใช้งานล่าสุด
5. ดูแลข้อมูลหลักและสิทธิ์การเข้าถึง

#### กระบวนการ Data Requests

1. จังหวัดสร้างคำขอข้อมูล
2. อำเภอหรือหน่วยงานที่เกี่ยวข้องกรอกข้อมูล
3. ระบบรวมข้อมูลกลับมาที่ส่วนกลาง
4. เจ้าหน้าที่ตรวจสอบและใช้ข้อมูลต่อ
5. ลดการส่งไฟล์ซ้ำ ลดการรวมมือ และลดความผิดพลาด

### ผลลัพธ์ที่ระบบสร้างได้

- Dashboard ภาพรวมเกษตรจังหวัด
- ข้อมูลแยกตามกลุ่มงาน
- ตารางจัดการข้อมูล
- กราฟสรุปแนวโน้มและจำนวน
- แผนที่แสดงพื้นที่และจุดเฝ้าระวัง
- Search result จากหลายตาราง
- คำตอบจาก AI ที่อ้างอิงข้อมูลในระบบ
- Public portal สำหรับเผยแพร่ข้อมูล
- ระบบจัดการคำขอข้อมูลจากพื้นที่
- audit และการควบคุมสิทธิ์สำหรับงานภายใน

### ประโยชน์ต่อเจ้าหน้าที่

- ลดเวลาหาข้อมูล
- ลดงานรวม Excel ด้วยมือ
- ลดการทำรายงานซ้ำ
- จัดการข้อมูลได้เป็นระบบ
- ค้นหาข้ามหลายตารางจากจุดเดียว
- ใช้ AI ช่วยสรุปข้อมูลจำนวนมาก
- เห็นสถานะข้อมูลของแต่ละกลุ่มงานชัดขึ้น

### ประโยชน์ต่อผู้บริหาร

- เห็นภาพรวมจังหวัดเร็วขึ้น
- ตัดสินใจจากข้อมูลจริงได้ง่ายขึ้น
- ตรวจสถานการณ์รายพื้นที่ได้
- เห็นข้อมูลเฝ้าระวัง เช่น ภัยพิบัติ hotspot สภาพอากาศ
- ใช้ dashboard ในการประชุม ติดตามงาน และวางแผน
- ลดการรอรายงานจากหลายฝ่าย

### ประโยชน์ต่อประชาชน

- เข้าถึงข้อมูลเกษตรจังหวัดง่ายขึ้น
- เห็นข้อมูลที่เปิดเผยได้โดยไม่ต้อง login
- เพิ่มความโปร่งใสของข้อมูลภาครัฐ
- เห็นโอกาสและเครือข่ายเกษตร เช่น วิสาหกิจชุมชน แปลงใหญ่ ท่องเที่ยวเกษตร Smart Farmer

### ประโยชน์ต่อหน่วยงาน

- มีฐานข้อมูลกลางที่ต่อยอดได้
- ลดการพึ่งพาไฟล์แยก
- วางมาตรฐานข้อมูลร่วมกันได้
- เชื่อมต่อ API ภายนอกได้ง่ายขึ้น
- รองรับระบบใหม่ในอนาคต เช่น data request, analytics, AI assistant, open data
- มีโครงสร้าง deployment ที่พร้อมเผยแพร่ผ่าน Netlify

### จุดเด่นเชิงนวัตกรรม

- รวม public portal และ internal dashboard ในระบบเดียว
- ใช้ Supabase เป็น backend กลางสำหรับ auth และ database
- ใช้ AI ช่วยถามตอบจากข้อมูลจริง
- ใช้ global search ค้นหาข้ามหลายตาราง
- ใช้ map และ chart ช่วยมองข้อมูลเชิงพื้นที่
- มี serverless proxy เชื่อมข้อมูลภายนอก
- รองรับ role และ department สำหรับงานราชการ
- มีแนวทางต่อยอด Data Requests เพื่อลดงานรับส่ง Excel

### ตัวอย่าง Before / After

#### ก่อนมีระบบ

- ข้อมูลอยู่หลายไฟล์
- ต้องถามหลายคน
- ทำรายงานช้า
- ดูภาพรวมยาก
- ประชาชนเข้าถึงข้อมูลจำกัด
- ข้อมูลพื้นที่รวมกลับส่วนกลางล่าช้า

#### หลังมีระบบ

- ข้อมูลรวมในฐานเดียว
- ค้นหาได้ทันที
- Dashboard สรุปภาพรวม
- AI ช่วยตอบคำถาม
- ประชาชนดูข้อมูล public ได้
- จังหวัดติดตามข้อมูลจากพื้นที่ได้เป็นระบบ
