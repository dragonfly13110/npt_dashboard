# โครงสร้างข้อมูลและฟิลด์เมทาดาทาสำหรับระบบ RAG

## 1. ขอบเขตชุดส่งออก

รุ่น 3.6 สร้างชุดข้อมูล RAG จากไฟล์องค์ความรู้ในหมวด 01-11 จำนวน **178 เอกสาร** โดยไม่รวมไฟล์ดัชนี แม่แบบ และรายงานระบบ เพื่อป้องกันข้อความบริหารโครงการปะปนกับคำตอบวิชาการ

ชุดส่งออกหลักอยู่ที่ `99_ระบบ/RAG_EXPORT/`

- `rag_chunks.jsonl` หนึ่งบรรทัดต่อหนึ่งชิ้นข้อมูล
- `rag_documents.csv` หนึ่งแถวต่อหนึ่งเอกสารต้นทาง
- `rag_evaluation.jsonl` ชุดคำถามสำหรับทดสอบการค้นคืนและกติกาความปลอดภัย
- `supabase_pgvector_schema.sql` โครงสร้างตารางตัวอย่างสำหรับ PostgreSQL และ pgvector
- `rag_export_metadata.json` สถิติ รุ่น และค่าแฮชของไฟล์

## 2. ฟิลด์ระดับเอกสาร

| ฟิลด์           | ความหมาย                                       |
| --------------- | ---------------------------------------------- |
| `document_slug` | รหัสเอกสารแบบไม่ซ้ำ ใช้เป็นคีย์หลัก            |
| `title`         | ชื่อเรื่องที่แสดงต่อผู้ใช้                     |
| `category`      | หมวดองค์ความรู้                                |
| `plant`         | พืชเป้าหมายเมื่อมีข้อมูล                       |
| `pest_type`     | โรค แมลง ไร สัตว์ศัตรูพืช หรือวัชพืช           |
| `status`        | สถานะและเงื่อนไขการตรวจสอบจากไฟล์ต้นทาง        |
| `source_year`   | ปีของเอกสารต้นทาง                              |
| `source_pages`  | หน้าพิมพ์ที่ใช้อ้างอิง                         |
| `source_path`   | ตำแหน่งไฟล์ Markdown ในคลัง                    |
| `risk_flags`    | ธงสำหรับบังคับตรวจฉลาก กฎหมาย หรือผู้เชี่ยวชาญ |
| `sha256`        | ใช้ตรวจว่าข้อมูลเปลี่ยนจากรุ่นก่อนหรือไม่      |

## 3. ฟิลด์ระดับชิ้นข้อมูล

ชิ้นข้อมูลแต่ละรายการมี `id` ไม่ซ้ำ เชื่อมกับ `document_slug` และเก็บหัวข้อย่อย ลำดับส่วน ลำดับชิ้น จำนวนอักขระ แหล่งที่มา และค่าแฮช

ข้อความใน `text` ถูกเติมชื่อเอกสารและหัวข้อย่อยไว้ด้านหน้า เพื่อให้ embedding มีบริบทมากพอแม้ชิ้นข้อมูลถูกค้นคืนแยกจากเอกสาร

## 4. ความหมายของธงความเสี่ยง

- `current_label_and_registration_check_required` ต้องตรวจฉลากและทะเบียนปัจจุบันก่อนนำไปใช้จริง
- `current_law_check_required` ต้องตรวจประกาศกฎหมายที่มีผลใช้บังคับล่าสุด
- `historical_legal_reference` เป็นข้อมูลกฎหมายจากเอกสารปี 2568 ไม่ใช่ฐานข้อมูลแบบเวลาจริง
- `requires_human_verification` มีจุดผิดสังเกตหรือเงื่อนไขที่ต้องให้คนตรวจ
- `safety_sensitive` เกี่ยวข้องกับพิษ ความปลอดภัย หรือเหตุฉุกเฉิน

## 5. คีย์ที่ควรใช้กรองก่อนค้นเวกเตอร์

ระบบควรกรอง `category`, `plant` หรือ `pest_type` เมื่อผู้ใช้ระบุชัด แล้วจึงจัดอันดับด้วย similarity วิธีนี้ลดโอกาสดึงอัตราของพืชหรือศัตรูพืชคนละชนิด

## 6. ข้อห้ามในการรวมข้อมูล

ห้ามรวมอัตราจากสูตรหรือความเข้มข้นต่างกัน ห้ามเปลี่ยนเครื่องหมาย `/` เป็น `+` และห้ามสร้าง PHI หรือสถานะทะเบียนที่ไม่มีในบริบท

## 7. การควบคุมรุ่น

ใช้ `document_slug`, `source_path` และ `sha256` ตรวจการเปลี่ยนแปลงก่อนอัปเดตฐานข้อมูล ควรลบหรือแทนที่ chunk เดิมของเอกสารนั้นทั้งชุดเมื่อค่าแฮชเอกสารเปลี่ยน
