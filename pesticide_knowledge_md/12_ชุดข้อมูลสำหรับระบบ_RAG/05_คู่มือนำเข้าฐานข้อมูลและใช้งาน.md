# คู่มือนำชุดข้อมูลเข้าฐานข้อมูลและเชื่อมระบบถามตอบ

## 1. ลำดับการทำงาน

1. เลือก embedding model และยืนยันมิติเวกเตอร์
2. รัน `supabase_pgvector_schema.sql` หลังแก้ `vector(768)` ให้ตรงโมเดล
3. นำ `rag_documents.csv` เข้าตารางเอกสาร
4. อ่าน `rag_chunks.jsonl` สร้าง embedding ทีละ batch
5. upsert chunk โดยใช้ `id` เป็นคีย์หลัก
6. สร้าง vector index หลังโหลดข้อมูลครบ
7. ทดสอบด้วย `rag_evaluation.jsonl`

## 2. การเลือก embedding

ควรเลือกโมเดลที่รองรับภาษาไทยและข้อความตาราง ทดสอบอย่างน้อยสองโมเดลด้วยชุดคำถามเดียวกัน อย่าตัดสินจากคะแนนภาษาอังกฤษหรือขนาดโมเดลเพียงอย่างเดียว

## 3. การนำเข้าข้อมูล

ใช้ `document_slug` เป็น foreign key และเก็บ metadata เป็นคอลัมน์เพื่อกรองก่อนค้นเวกเตอร์ ค่า `risk_flags` ควรเป็น array เพื่อให้ระบบบังคับคำเตือนหรือส่งต่อให้คนตรวจได้

## 4. การอัปเดตรุ่น

เปรียบเทียบ `sha256` ระดับเอกสาร หากเปลี่ยนให้ลบ chunks เดิมของ `document_slug` นั้น แล้วสร้าง embedding ใหม่ทั้งหมด ห้ามผสม chunks คนละรุ่นในเอกสารเดียวกัน

## 5. Hybrid search

ควรใช้ full-text หรือ keyword search ร่วมกับ vector search เพราะชื่อสามัญ สูตร เช่น `50% WP` และรหัสกลุ่มต้องการการจับคู่ตรง ส่วนคำถามภาษาพูดต้องอาศัย semantic search

## 6. การจัดอันดับเพิ่มเติม

เพิ่มคะแนนเมื่อ `plant`, `pest_type`, ชื่อสามัญ สูตร หรือกลุ่มตรงกับคำถาม ลดคะแนนเมื่อพืชต่างชนิดหรือเป็นเพียงดรรชนีชื่อสาร

## 7. การแสดงคำตอบ

ส่วนอ้างอิงควรแสดง `title`, `source_pages`, `section_heading` และ `source_path` ส่วนธงความเสี่ยงควรแปลงเป็นข้อความเตือนภาษาไทยที่ผู้ใช้เข้าใจได้

## 8. เงื่อนไขก่อนเปิดใช้จริง

- Recall@5 ผ่านเกณฑ์ที่กำหนดกับคำถามจริง
- ไม่พบการย้ายอัตราข้ามสูตรในชุดทดสอบ
- คำถามทะเบียน กฎหมาย และ PHI ถูกควบคุมตาม guardrail
- มีระบบเก็บ feedback และ trace ว่าแต่ละคำตอบใช้ chunk ใด
- มีผู้รับผิดชอบทบทวนข้อมูลเมื่อฉลากหรือกฎหมายเปลี่ยน
